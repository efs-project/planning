## 1. Attacks on [[access-delegation]]

### A1. The approval sweep cannot answer a path read — record-mode collides with §7.2 — SERIOUS, fix exists, one sentence

**The claim attacked:** §3.2's record-mode ("the curator counter-signs specific claims — an approval TAG targeting the claimId (or the curator re-PINs / places Bob's DATA under the canonical path themselves)"), and its load-bearing uses: the blessed remove-member sweep (§3.2), 8.A.5 ("her old good work returns via approval sweep"), 8.B.4, 8.C.4 ("Kept work persists under Alice's approval slots").

**The attack.** Walk the read. Bob is ejected; Alice ran the closing sweep as approval-TAGs (`(Alice, approvedDef, claimId_bob_plan)`). A reader now resolves `/acme/projects/plan.md` under lens `[Alice]`:

- Position `(Alice, plan.md-placement-slot)` — Alice never placed `plan.md`; PROVEN-ABSENT → fall through.
- Bob is not in the lens. Resolution returns AbsentEverywhere. **The approved work does not appear.**

The approval TAG lives at a *different slot* than the key being resolved. To make it count, the resolver would have to: enumerate candidates at the placement key (discovery), then point-read Alice's approval per candidate. That is enumeration **inside slot resolution**, which §7.2 forbids in exactly those words ("Discovery output MUST NOT enter slot resolution … or satisfy a PROVEN-ABSENT check"). The prohibition is not incidental — it carries the venue-completeness and spam-shaped reasons the whole discovery doctrine rests on. So as written, the sweep preserves Bob's work for *accumulation-shaped* views (comment lists, feeds — the §9.A verified-section pattern, where enumeration is the read) but **not for path-shaped views** — and the walkthroughs (8.A.5, 8.C.4) are all path/folder-shaped.

**Also broken downstream:** the SDK affordance "Keep their 47 contributions in the project view? [approve all]" silently produces the non-working variant if implemented as approval-TAGs.

**Minimal fix (inside the design, and the report half-says it already):** bless the sweep's mechanics per view shape. For path/folder views the sweep is **curator re-placement**: Alice PINs the kept claimId/dataId at *her own* placement slot per key (47 keys = 47 PINs, one envelope, one signature — same ceremony cost; and it is the strongest form: GATE consumers read curator slots directly, the §4.6 countersign rung). Approval-TAGs remain correct for accumulation views only. Do not invent an "approval-aware resolution mode" — it would need a §7.2 carve-out for no gain over re-PIN. One paragraph in §3.2 + the SDK verb spec; a conformance vector ("swept-then-ejected member's file resolves at the curator's position, not the member's").

**Severity: serious** — the lane's flagship UX answer to its own #3 failure mode, specified in the variant that doesn't resolve. Fully repairable with zero new surface.

### A2. Delegated resolution enumerates the delegate set inside resolution — needs the completeness rule it doesn't state — SERIOUS, fix exists

**The claim attacked:** §4.3 resolver rule 2: "enumerate the team's LIVE `act` slots (bounded point reads / small enumeration under the team's container)."

**The attack.** Point reads require already knowing each delegate's address (the slot key is `(team, act, delegate)`). Discovering *which* delegates exist is an enumeration — discovery-shaped, and §7.2 again: "nothing enumerated ≠ proven absent." On any venue that is not the team's live home chain, the enumerated act-set can be silently incomplete: delegate E4's grant hasn't replicated → the resolver doesn't know to probe `(E4, K)` → the reader serves a lower-precedence delegate's record or absence, **with no UNKNOWN anywhere** — a data gap silently changing who speaks for ACME. That is the anti-fallthrough disease (a gap converted to a trust transfer) reintroduced one level up, at the *author-set* rather than the position. The report's own failure-mode register (§11.7, delegation-dialect fork: "divergent trust between conforming clients is a security event") is exactly what an unstated enumeration rule produces.

**Why it's fixable inside the design:** the completeness machinery already exists for a *single author's own set* — the checkpoint. A team's act claims all live under the team's container, one author; checkpoint non-inclusion gives "no grant/revoke beyond seq N" — an as-of-N-complete delegate set. **Minimal fix (Durable, read-lens-spec text):** the delegate-set read is an *author-scoped, checkpoint-bounded enumeration* graded like any read — HOME-LIVE = complete; AS-OF(N) = complete-through-N, age displayed and inherited by every expansion result; no checkpoint coverage = the expansion itself is UNKNOWN and delegation-aware resolution MUST degrade to team-authored-claims-only (never partially expand). Same one-sentence rule the persona stitching in [[wallet-and-actions]] needs and also never states (flag to that doc too — "all bidirectionally-LIVE linked personas" has the identical hole). Alternative same-cost fix: the team maintains the delegate roster as a single LIST so the set is slot-shaped; either works, pick one and vector it.

**Severity: serious** — a silent-wrong-answer class on replicas, in the one place the report itself says dialect divergence is a security event. No new rows; one normative paragraph + one vector (delegate grant present at home, absent at replica → replica must not serve a different delegate's claim unlabeled).

### A3. Grant expiry + LIVE-only resolution = org-wide history wipe at every renewal lapse — the 90-day doctrine is a footgun without P1 — SERIOUS, fix exists

**The claim attacked:** §8.B.6 celebrates 90-day act expiries ("a live audit … TOFU decay, deliberate") and §11.11 prescribes expiry-by-default on grants. Meanwhile §4.3 resolver rule 2 expands only **LIVE** act slots and §3.3/§4.3 rule STALE-act = not-authority, fail closed.

**The attack.** Trace the lapse. ACME's re-grant ceremony slips three days past the 90-day expiry. Every act claim goes STALE → delegation resolution stops → **all delegate-authored content under /acme/blog — years of history, not just new writes — leaves every delegation-aware view** until the ceremony runs. And re-granting does *not* trivially restore it under window semantics: the new grant's validity window starts at its assertion; the old records' admission times predate it. The coherent window rule is "record in-window of the grant that covered its admission" — which (i) requires `admittedAt` (P1), (ii) requires the resolver to test records against *superseded/expired* grants, not just LIVE ones — a rule the report never writes (rule 2's "LIVE act slots" contradicts it), and (iii) without P1 cannot be offered at all (the report's own §3.3 conclusion), leaving the flicker as the *default shipped behavior* of the lane's *recommended default configuration*. The failure-mode register names stale-grant creep (§11.11) but not its inverse, which is worse: the defense against creep **causes** the wipe.

**Minimal fix (inside the design):** three coupled sentences. (1) Delegated resolution distinguishes *authorizing new admissions* (requires a LIVE grant) from *honoring past records* (a record is honored if some grant by the team covered it — in-window per admittedAt where P1 exists). (2) Without P1, expiry-by-default on act grants MUST ship together with the record-mode/countersign rung for anything meant to outlive the grant (the report's "windows for convenience, sweeps for permanence," promoted from guidance to a conformance MUST for the 90d default). (3) Add the failure mode to §11: **renewal-lapse view wipe** — and make the SDK re-grant verb warn at T-minus. This also sharpens the P1 weight the report already sends to [[freeze-gates]]: P1 is load-bearing not just for windowed *membership* but for the lane's own recommended expiry doctrine.

**Severity: serious** — an operational cliff baked into the blessed pattern; repairs are Durable text + doctrine, no freeze surface.

### A4. Windowed authorization is venue-divergent because replication reorders `admittedAt` — survivable, name it

`admittedAt` is per-venue. A delegate record admitted at home in-window, replicated to venue V *after* the revoke/expiry replicated there, gets a V-local `admittedAt` outside the window: **the same record is authorized at home and unauthorized at V, permanently.** The report says "venue-relative" (§3.3.1) but not this inversion, which is the concrete thing an app developer will hit. GATE is unaffected (GATE consumes LIVE grants only). **Fix:** one doctrine sentence — window tests anchor on the *earliest known* `admittedAt` (the verify-time-model already anchors `claimedAt`-falsifiability on "earliest/home admittedAt" — reuse that anchor), render venue-qualified otherwise; and the portable answer remains record-mode. Survivable: an honesty/spec gap, not a wrong mechanism.

### A5. "STALE membership: INTERACTIVE may label" is ambiguous exactly where the guest walkthrough needs it — survivable, one sentence

§3.3 says a STALE membership entry "MUST be treated as not-in-lens for GATE resolution … INTERACTIVE may label." Label *what* — the guest's still-rendered content, or the historical fact of membership? If the former, the guest's write access **effectively continues past expiry in every INTERACTIVE view** (read-lens-spec's INTERACTIVE default for STALE content is label-and-render — a reader could reasonably apply it to the membership claim), and "guest for a week" is really "guest forever, with a label," which is not what 8.C.4 promises ("G's contributions leave the live view"). **Fix:** the lens-object spec rules it: authority-bearing claims (membership entries, act grants) that are STALE are **excluded from resolution in ALL contexts**; INTERACTIVE may render the historical membership in history views only. State why the asymmetry with content-STALE is right: content-STALE renders because the archive is the product; authority-STALE excludes because expired authority never authorizes — two different objects of the same grade. Vector it (it will fork clients otherwise).

### A6. "An EQUIVOCAL team author poisons the act claims themselves ⇒ delegation resolution stops" is over-broad as written — survivable

EQUIVOCAL is per-`(author, seq)` (P6), not per-author. One accidental SeqCollision by the org key (multi-custodian org keys are the *most* likely self-equivocators — exactly P10's device-bit problem, at the identity with the highest blast radius) forks one seq region; only act claims *at that region* grade EQUIVOCAL. The report's sentence reads as "any team duplicity halts all delegation" — which would be a new rule beyond the grade algebra and a lens-wedging lever (one stray collision = org-wide outage). **Fix:** re-word to the per-claim semantics (an EQUIVOCAL *act claim* is not-authority, fail closed for that delegate; team duplicity elsewhere is ordinary evidence, surfaced not obeyed), and send the org-key device-bit allocation warning to the §4.6 custody rungs.

### A7. Assigned question — does delegated authorship at year-0 secretly require the KEL? **No** — with three degradations, two admitted, one under-admitted

Adjudicated mechanism by mechanism: grant/re-scope/revoke (prospective), scope evaluation (parent-walk + kind/key words — all derivable from record + policy; parent-walk is upward via `getObject` parent linkage, bounded), dual attribution, GATE two-step, blast-radius containment — **all work with bare EOAs and no kernel change. The construction does not smuggle in the KEL.** What silently degrades: (1) thief-after-N partition — admitted loudly (§5 residual, §8.B.5); (2) org-key rotation — admitted; (3) the window story — *under*-admitted: as A3/A4 show, windows aren't merely "unavailable without P1," their absence turns the lane's own default doctrine into a cliff, and with P1 they are venue-divergent without an anchoring rule. Also worth one line in §4.6: threshold custody of a single secp256k1 org key via MPC-TSS produces standard signatures and is a real year-0 m-of-n *custody* answer (identity am.3's "no m-of-n authorship" is about protocol-native thresholds); the report's "hardware/threshold custody" gestures at it without naming it.

### A8. Guest-for-a-week revocation reality — holds, with two honesty additions for the privacy variant — survivable

The public-folder story survives attack: per-reader eventual propagation is stated (§3.1), deny + revoke mid-week both work at next refresh, stale-pinned-lens readers keep serving G with venue-currency labels (correct and honest), the back-date defusal via admittedAt is right and correctly gated on P1. The **capability variant** (8.C.6) is missing two permanent facts that belong in §6.3's cost list: (i) a past member can not only *read* old content forever — they can **write into the salted namespace forever** (the cap = addressing = candidacy; §6.2 says this but §6.3's removal-cost list omits it), so remaining members' clients will enumerate the leaver's graffiti in DISCOVERY views for the life of the salt; (ii) **any past member can de-blind the namespace permanently for everyone** by publishing the salt — one leaver, no defense, and re-salting protects only the future tree. Both are inherent to caps (Tahoe has the same properties); the report's own standard ("state this in every private-folder UI") demands they be listed. Also import the deletion lane's KEK/lazy-re-key interval honesty ("removed; N files pending re-key") so the two lanes' removal stories can't drift.

### A9. Sweep and roster costs at scale — survivable, price it

The sweep is priced as "one ceremony" but never in gas: a departing employee with 10k claims ≈ 10k re-PINs ≈ the mount lane's compaction order of magnitude (~a quarter-billion gas class). Not a break — one signature, chunked admission via `submitSubset`, monotone — but the access report should cite the mount lane's costing so "the SDK offers the sweep" doesn't imply it's free at org scale.

### A10. Assigned hunt — does declared-gone W4 (write-exclusion) hide an app that truly needs it? **Empty, concurring with [[consistency-atomicity-longtail]] §2 — with one residual, named**

Independent candidates tried: licensed/regulated-writer systems (medical, land registry) — correctness lives in the reader's lens/curator, re-homed, holds; court-ordered exclusion — read-side deny + policy pass, holds; anti-squat — per-author slots, vacuous; scarce names/money — chain-layer by mission, holds (consistency lane, same verdict); private-folder integrity — lens rejects, holds (A8's residual is visibility of graffiti, not authority). **The one thing exclusion also used to buy that nothing re-homes: enumeration-cost protection.** A whale can flood one hot container's discovery index at their own gas; every entry is lens-rejected at render, but INTERACTIVE listings and the delegate/roster enumerations of A2 pay page-walks through the spam (256/page). That is a *liveness/cost* asymmetry, not a correctness or permission gap — the accepted doctrine ("spam absorbed at writer's gas, contained per container") prices it, and the indexer lane filters it — but it is the honest answer to "why did POSIX have `w` bits on directories" and should appear in the W4 disposition row so the declared-gone verdict states what was actually given up. **Disposition confirmed: W4 GONE, correctly** — with the enumeration-cost residual added to the row.

## 2. Attacks on [[namespaces-mounts-federation-move]]

### M1. R-M1's evaluator is undefined: "resolves" is a nested lookahead with unstated budget, granularity, and result-classes — SERIOUS, fix exists

**The claim attacked:** §3.2's three arms, keyed on "the resolution target at the old ids resolves / comes up AbsentEverywhere for the suffix segment being resolved."

**The attacks, stacked:**

1. **Lookahead recursion.** Deciding arm 1 vs arm 2 at moved node F requires *attempting full suffix resolution* under old ids. If the suffix crosses another moved node G (nested independent moves — F→F′ while F/G→H), the lookahead recurses: the arm-decision at F depends on an arm-decision at G. Coherent, but nowhere stated; naive implementations will test only the *next* segment or only the leaf, and they will disagree on any partially-compacted tree. (Note: intermediate segments are pure keccak derivations — TAGDEFs are permanent and unrevocable, so "the folder still exists at old ids" is always true; the only testable absences are claim slots, i.e. placements and edges. The rule must say the lookahead tests the *claim-bearing* positions, not the derivable spine.)
2. **Budget accounting inside the lookahead.** Follows consumed while *probing* the old-id suffix (e.g. G's movedTo inside the lookahead) — counted against the same budget as the outer walk? Refunded if arm 2 fires and the walk retargets? Under the per-segment re-cut (§3.7), does the lookahead's probe of segment j charge segment j's budget? Unstated; two implementations diverge on deep trees near the budget — exactly the "rule nobody wrote" class this lane exists to close.
3. **Result-classes.** The lookahead can return Denied, Stop(STALE), Stop(EQUIVOCAL), Unresolved(UNKNOWN) — the arms name only PRESENT/PROVEN-ABSENT/UNKNOWN. Arm 2's "AbsentEverywhere only" wording *implicitly* handles this correctly (Denied/STALE are not AbsentEverywhere → no retarget — vital: a deny hit converting into an arm-2 redirect would let an advisory subtraction hand the name to the move target, the deny-shaped twin of the trust-inversion rule in §3.4.1), but implicit is not a spec. One misreading — "content not servable here, follow the crumb" — reintroduces the inversion.

**Minimal fix:** write R-M1 as an evaluator, not three arms: *at a moved node, run the ordinary §3 resolution of the remaining suffix under the old ids (nested moves recurse; all follows charge the shared budget with no refunds); retarget to the movedTo target iff that resolution returns AbsentEverywhere; every other outcome — Resolved, Denied, Stop(any), Unresolved — is served/propagated in place with the breadcrumb.* Plus vectors: nested-move, Denied-not-retargeted, STALE-not-retargeted, UNKNOWN-stops, budget-exhaustion-inside-lookahead. This lands in the same §9-3 Etched-table re-wording window the lane already flagged — the vectors cut for the conditional-follow column MUST include these cases, or the frozen vectors bake in the ambiguity the re-wording was meant to kill.

**Severity: serious** — the lane's #1 repair, correct in intent, unimplementable interoperably as written.

### M2. The union walk needs spine-set semantics; the "current parent" fix is singular where the walk is plural — SERIOUS, fix exists

**The claim attacked:** §6's one-sentence classifier fix ("define *current parent* as the post-follow container id") and R-A2's per-segment layering.

**The attack.** After a graft (or a lazy move's continuity symlink), a segment has **two** live derivation parents — the direct (M-derived) id and the grafted (T-derived) id — and R-A2 keeps both alive for listings and for resolution-until-absent. So "the current parent" is not an id, it's an ordered *set* of ids (a spine-set), and every subsequent NAME segment derives under **each** spine. The report's own T5 (split-spine: legacy content at old ids, Carol's new writes at new ids, both accruing) proves the plural case is the *steady state* of every lazy move, not a corner. Questions the one-sentence fix leaves open, on which conforming resolvers will fork: does serving segment *i* from the direct layer bind segment *i+1* to the direct spine only (killing grafted grandchildren whose intermediate name exists in both layers — and intermediate names *always* exist in both layers, because TAGDEF spines are derivable, not claimed)? Are the layers ordered per-segment or once at the graft? Does a second graft *inside* the grafted layer extend the set? **Minimal fix:** state the walk as carrying an ordered spine-set — seeded `[direct]`, extended (not replaced) at each graft in R-A2 order, every NAME deriving under all spines, claim-bearing reads evaluated in spine order with first-non-absent winning, UNKNOWN in any consulted spine stopping (the layer-blind rule, kept) — with a dual-parent vector (name claimed in both layers; grandchild claimed only in grafted layer under an intermediate name present in both). This is Durable read-lens-spec text; without it the union convention (§2.4) and every lazy move fork clients silently. The classifier itself survives (rule-1 prefixes and root classification untouched) — the break is confined to the rule-3 "current parent," which is fixable exactly there.

### M3. The listing rules are asymmetric and one of them, applied literally, makes a folder with one dead graft unlistable — SERIOUS (for the layer-blind listing case), fix exists

Three listing gaps, one rule each:

1. **Layer-blind UNKNOWN applied to listings.** R-A2: "UNKNOWN at either layer STOPS (anti-fallthrough is layer-blind)." Right for single-key resolution (never substitute across a gap). Applied to *listings*, it means one graft whose target author has no checkpoint coverage at this venue turns the whole folder — including its perfectly-known direct layer — into a stop. Listings are INTERACTIVE surfaces; the honest render is *partial-with-labeled-gap*: direct-layer entries + "grafted layer unknown at this venue (homed at Y)" (composing with the §2.6 federation UX rule), never a bare stop and never a silent direct-only render. GATE consumers don't consume listings at all (§7.2 / R-A3), so nothing machine-facing weakens.
2. **Old-node listings during/after compaction.** T5 handles listings at the *new* node (graft-union). At the *old* node mid-compaction, compacted children have revoked placements → they simply vanish from the old listing one by one, with only a node-level breadcrumb; the report's per-child chunking prevents the void but not the shrinking-folder render. Post-full-compaction the node-level retarget (arm 2) heals it. **Fix:** one sentence — a listing at a node with a LIVE `movedTo` SHOULD render the union-with-breadcrumb (old remainder + target's listing, labeled), mirroring R-A2; or at minimum must render the crumb *as a listing banner*, not a per-file surprise.
3. **Same-name cross-layer double-listing.** R-A2 says listings "union both layers, each entry labeled" — but single-name resolution is direct-first-shadowing. A name claimed in both layers therefore *lists twice and resolves to one*: the listing shows an entry the path walk will never serve. OverlayFS (the cited source) shadows per-name. **Fix:** listings collapse same-name collisions to the resolution winner with a "shadows N grafted entries" affordance; the WHITEOUT pin (§9-8) then has a well-defined thing to mask. Vector: name in both layers → one listed entry, marked.

### M4. R-M1 is an availability regression on partial venues — survivable, must be named

Under the old (broken) unconditional follow, an old deep link through a compacted move resolved on *any* venue holding the target's claims. Under R-M1, arm 2 requires **PROVEN-ABSENT at the old ids** — i.e. home-total-state or checkpoint non-inclusion for the lens authors. On a cherry-picked or checkpoint-less replica, old links through compacted moves now return Unresolved where they used to (incorrectly but usefully) serve. This is the right trade — absence-with-a-bound is the only honest license to redirect — but it is a real regression class ("my old links work at home and die on mirrors") and the report doesn't name it. **Fix:** add to the register (FM-move-venue-degrade) + one doctrine line: movers who compact SHOULD publish a fresh checkpoint in the compaction envelope (it makes their own old links resolvable off-home — self-interested, so adoption is natural), and the SDK compaction verb includes it.

### M5. Grafting to someone else's tree requires minting their TAGDEF chain at your venue — survivable, unstated admission step

REF-edge rows keep typing/existence checks (codex-kinds: "typing/existence checks survive as frozen table rows"; envelope master invariant: missing-dependency reverts are the retryable class). A graft PIN targeting `tagId(B/photos)` therefore needs that tagId **registered at the admitting venue**. If B's TAGDEFs never replicated there, Alice must mint B's path chain herself — legal (TAGDEFs are unowned Schelling points, parents-first), bounded (≤ depth mints), but a real choreography step T2 and the lazy-move walkthrough silently assume away (the lazy move mints the *new* chain, §3.3 step 1 — the cross-author graft case is the unstated one). **Fix:** one sentence in §2.3 + the SDK graft verb mints the target chain parents-first; note the cost. Also fold into §9-6's table-cell confirmation (legal targetKinds) so both halves of "can I admit this edge here" are checked before the table freezes.

### M6. Visited-set semantics: global-per-resolution false-positives on legitimate diamonds — survivable

With the union convention (§2.4), one resolution can legitimately reach the same node along *different union branches* (two overlay targets sharing a common library subtree — the Plan-9 `/bin` case the convention exists for). A **global** visited-set marks the second branch's arrival a "cycle" → UNRESOLVABLE on a healthy, acyclic DAG. The set must be **per-walk-path** (detecting true cycles: a node revisited along one chain), and union-edge follows must be *in* it (the report counts their budget but never says they enter cycle detection). **Fix:** one sentence + a diamond vector. Interacts with §3.7's re-cut — specify both in the same constant-shape amendment so they're vectored together.

### M7. The graft's other blade: the target controls content under the mounter's prefix — survivable, register line

FM-mount-shadow covers the mounter shadowing the target (fixed by R-A1). The converse surface is unnamed: after Alice grafts `/A/shared/photos → B/photos`, **B serves arbitrary future content under A's perceived namespace** — R-A1 *guarantees* B wins there — and A's audience browses it with A's prefix in the address bar. That is what mounting means (delegation of a subtree), it is disclosed by the chip and the "via graft at M" label, and the phishing shape is §4.2's authorship-boundary discipline extended — but it belongs in the register with the mitigation stated (grafts are live trust edges; the pinned/citation form (§2.5) is the no-trust alternative; the SDK graft verb says so). One line, one hostile vector (target swaps content post-graft; chip renders B, not A).

### M8. Split-spine residuals: cross-venue duplicates and doubled enumeration cost — survivable

Per-child compaction chunking prevents the mid-move void *at the admitting venue*; a replica that received the new-PIN chunk but not yet the old-REVOKE chunk lists the child in both spines (two LIVE placements, same content) — transient, converges, but clients should dedupe by target dataId in the M3 collapse rule (which fixes this for free).
