## 0. The ruling in one paragraph

**ZK helps EFS at the edges and poisons it at the core, and the line between the two is exactly the line between *proving things about the graph* and *hiding the graph*.** Narrow-yes: ZK membership and selective-disclosure proofs are **sidecar artifacts** — proofs carried in record bodies and verified client-side (portable), or verified by redeployable sibling contracts per chain (composable) — that let EFS say "a member of set S asserts this," "the caller is on the list without the list being public," and "this ciphertext has property P," none of which encryption alone can say. Broad-no: shielded-pool-for-files (Zcash/Aztec-for-EFS) forfeits verify-don't-trust, hyperlinkability, and on-chain composability simultaneously — it is a different product, and the user who truly needs a hidden graph has a zero-infrastructure escape hatch already expressible in v2 (§2.4). The freeze-sensitive result, adversarially checked in §4: **the already-reserved surface (digest-shaped author space, deferred algoTag minting pattern, OPAQUE targets, permissionless key-TAGDEFs, VAL ≤ 8192, EFSBytes, peer-kernel succession) is sufficient for every narrow-yes item to ship post-freeze — ZK demands no new kernel row, no new derivation domain, and no new envelope byte.** What it demands is one frozen sentence of doctrine (ZK state never gates admission, §2.3/§4-R1) and one honest correction to the corpus: the "Fileverse-proven ZK membership" premise in privacy.md is mislabeled — their shipped library contains **no zero-knowledge proofs at all** (§1.2). The production-proven private-membership reference is Semaphore, and the production-proven Fileverse mechanism is an OPRF-server pattern that EFS's random-occurrence-key ruling already dominates on EFS's own threat model.

## 1. Research grounding (PART A)

### 1.1 Semaphore v4 — the reference for anonymous set membership (VERIFIED, primary spec + docs)

Read: the [Semaphore V4 specification (zkspecs #3)](https://github.com/zkspecs/zkspecs/blob/main/specs/3/README.md), [contracts docs](https://docs.semaphore.pse.dev/technical-reference/contracts), [v4.0.0 release notes](https://github.com/semaphore-protocol/semaphore/releases/tag/v4.0.0), [what-is page](https://docs.semaphore.pse.dev/).

- **Identity:** EdDSA on Baby Jubjub. Private key → secret scalar (Blake1, replacing RFC 8032's SHA-512) → public key point; the **identity commitment = Poseidon(publicKey)** is the public value that goes in groups. Note: this is **not** a secp256k1 EOA key — an EFS author cannot drop their signing key into Semaphore; a separate credential key is required (consequence in §2.7).
- **Groups:** LeanIMT — binary incremental Merkle tree, dynamic depth, Poseidon nodes, always balanced; add/update/remove supported. v4.0.0 gas: LeanIMT insert ~143k gas (16 leaves) to ~252k (1024 leaves), 35–45% cheaper than v3's IMT; batch `insertMany` 54–78% cheaper again.
- **Circuit:** private inputs = secret scalar + Merkle siblings/indices/depth; public = message (bytes32) + scope (bytes32); outputs = Merkle root + **nullifier = Poseidon(scope, secret)**. The message is bound by a dummy-square constraint. One proof = "a member of the group whose root is R sent message M under scope S," with at most one un-linkable action per (member, scope) — a second proof under the same scope reveals *reuse* (same nullifier), never *identity*.
- **Proving system:** Circom + **Groth16** + snarkjs; MAX_DEPTH 1–32 compiled per depth; **trusted setup ceremony completed 2024-07-13 with 400+ participants** ([docs](https://docs.semaphore.pse.dev/)). `SemaphoreVerifier.sol` is an extended snarkjs Groth16 verifier holding vkeys for each depth; `validateProof` checks nullifier-unseen, verifies, stores nullifier.
- **Maturity mid-2026:** v4 stable and shipped; the 2024 roadmap evaluated Noir ports and a HashCloak team [proposed a Semaphore-Noir implementation (2025-02)](https://github.com/orgs/noir-lang/discussions/7523) with UltraPlonk/UltraHonk verifiers; I could not verify from primary sources that a Noir version has been merged into semaphore-protocol as of 2026-07 (see Confidence).

### 1.2 Fileverse `zk-granular-permissions` — READ THE CODE: there is no ZK in it (VERIFIED, every source file)

Read: [repo](https://github.com/fileverse/zk-granular-permissions) — README.md, package.json, `src/granular-permissions/{index,utils,merkle-tree,encryption,types}.ts`, `src/voprf/index.ts`, contract ABI. Version 0.0.1; deps: `@cloudflare/voprf-ts`, `@openzeppelin/merkle-tree`, `@fileverse/crypto`, `viem`.

**What the code actually does** (the whole mechanism, verified line-by-line):

1. Build an **OpenZeppelin `SimpleMerkleTree`** over `keccak256(identifier)` leaves, where identifier = email string or wallet address (`merkle-tree.ts`).
2. For each member, compute `proofBytes` = the *plain* concatenated Merkle-proof siblings; `treeKey = keccak256(proofBytes)` indexes that member's entry (`utils.ts:createEncryptionKeyEntry`).
3. Send **blinded** `proofBytes` to a **vOPRF evaluation server** (Cloudflare `voprf-ts` client; `voprf/index.ts` does blind → POST to `evaluationUrl` → finalize). HKDF the OPRF output into an AES key (`encryption.ts`).
4. AES-encrypt the file key / agent key / comment key under that OPRF-derived key; JSON = { merkle tree encrypted under a file-specific `secretKey`, owner permission map encrypted via callback, the `treeKey → encryptedKeys` map }; upload to IPFS; store the CID on the `FileversePermission` contract along with **plaintext per-file agent smart-account addresses + roles** (`initializeFilePermission` args, `utils.ts`).
5. Access = recompute your own Merkle proof → re-run the vOPRF → derive the same AES key → decrypt your wrapped keys.

**There is no circom, no noir, no snarkjs, no proof object, no verifier — anywhere in the dependency tree or source.** The Merkle "proof" is used as a *key-derivation input*, never verified by anyone; membership is enforced by *ability to decrypt*, exactly like EFS's keyWrap. The README's own acknowledgments describe the work as "inspired from the great vOPRF R&D done by [PSE] on the semaphore protocol and zkemail" — inspiration, not usage. The "zk" in the package name is branding.

**What the mechanism actually contributes** (and it is genuinely clever): the vOPRF server's secret key makes the identifier→key map **impossible to brute-force offline**. Emails and addresses are low-entropy; a purely hash-derived key map would fall to a dictionary attack on the public IPFS JSON. Forcing every derivation through a rate-limitable server turns an offline dictionary attack into an online, rate-limited one — the Privacy Pass / OPAQUE pattern applied to file sharing. **The price is a liveness-and-privacy-critical server**: if the evaluator is down, nobody can derive keys (walk-away test fails without it — note their own "Walkaway" tool must ship the OPRF key or a bypass, unverified which); the evaluator sees *when* and *how often* accesses happen (though not who/what, since inputs are blinded); and the operator can silently deny service (a soft read-ACL — precisely the thing EFS's model refuses to have). PSE's [vOPRF-ID](https://github.com/privacy-ethereum/vOPRF-ID) is the hardened version of the same idea — **threshold MPC nodes** replace the single server, with client ZK circuits (Noir) for verifiability; explicitly early-stage ("implementation will be updated and improved").

**Residual leakage of Fileverse's shape, adversarially read:** (i) per-file **agent addresses and roles are on-chain in plaintext** — recipient privacy holds only if agent keys are fresh per file and never linked; (ii) `treeKey = keccak256(plain Merkle proof)` is public in the IPFS JSON — an adversary who can guess the **entire member set** can rebuild the tree and confirm it (a guess-the-whole-set oracle; much harder than the O(1) per-member oracle EFS's red team killed in A1, but not nothing); (iii) their README/[FAQ](https://fileverse.io/faq) is honest that permission existence and update timing leak. Their FAQ separately claims "vOPRFs and zero-knowledge Semaphore contracts" keep collaborators anonymous — I found no Semaphore usage in this library; if it exists it is in an unpublished or different repo (could-not-verify; flagged).

**Correction to privacy.md §7 (with cause, per the honesty norms):** the sentence "their `zk-granular-permissions` uses vOPRF-blinded identifiers + encrypted Merkle membership to hide who your collaborators are on a public chain, today, in production" is *mechanically* accurate but the doc's framing of it as "ZK-membership['s] … live production instance" (§9.2, §10) is wrong — it is production evidence for **OPRF-hardened key distribution**, not for zero-knowledge membership proofs. The live production references for actual ZK membership are Semaphore deployments (Worldcoin's World ID uses Semaphore-style inclusion proofs at very large scale — PLAUSIBLE, not re-verified this round) — not Fileverse.

## 2. Structural rulings (PART C)

### 2.1 Proof portability: where proofs live and where they verify (C-1)

Records are chain-free signed artifacts; verifier contracts are chain-bound bytecode. The reconciliation, ruled:

- **RULING P-A: proof-in-record-body, verified client-side, is the portable form and the default.** The proof rides as bytes inside the signed claim body (VAL tail or EFSBytes REF for large proofs — see size note below). What is standardized is **the circuit**: its constraint system, verification key, and vector suite, published *on EFS* as permanent DATA (contentHash-pinned) and named by a registry convention — the standard is the circuit, not any contract. Any reader on any chain, or offline from a bundle, re-verifies the proof against the pinned vkey exactly as they re-verify a secp256k1 signature. This is verify-don't-trust preserved: the proof is just a second kind of self-verifying evidence.
- **RULING P-B: proof-at-gate is per-chain sibling verifier contracts, never kernel surface.** When a *contract* must consume a proof (use-case d), verification happens in a redeployable, non-Etched sibling (the EASExporter/view-contract class), which reads any EFS inputs it needs (group-root claims) through the frozen read ABI (`getSlot`/`getClaim`) and keeps its own nullifier state. Sibling verifiers are honestly chain-bound and honestly replaceable — a broken circuit means "redeploy the sibling," not "the archive is wrong."
- **VERIFIED against the kernel doc: the Etched kernel can never embed a verifier.** Three independent grounds. (1) Frozen-forever + circuit risk: the kernel has "no admin, no proxy, frozen forever" (codex-kernel adopted core); SNARK soundness bugs are a real, recurring class (the Zcash 2018/2019 counterfeiting-bug class; verifier-contract bugs in production rollups — PLAUSIBLE class, well-documented history), and a frozen broken verifier = forged records admitted forever with no fix. This is the same argument that rejected the pre-committed KEL registry address: "unwritten code cannot be hash-pinned." (2) The master admission invariant: "nothing may permanently reject what another kernel could accept" — proof-conditioned admission couples the admitted set to circuit choice and to chain-local precompile/gas behavior (BN254 pairing precompiles were repriced by Pectra in 2025 — [verified via gas-cost sources above](https://xn--2-umb.com/22/groth16-tweaks/)); two kernels with different verifier vintages would permanently diverge. (3) Cost: admission is ~22–27k gas/record on the spine; a Groth16 verify is ~200–350k — proof-checked admission would 10× the hottest write path for a feature most records don't use. **All three converge: ZK may add evidence to records and gates; it may never gate kernel admission.** (This becomes the frozen sentence R1 in §4.)
- **Size note (freeze-relevant fact, no change needed):** VAL ≤ 8192 bytes comfortably carries Groth16 proofs (~128–256 B) and 32-byte roots/nullifiers/commitments inline; it does **not** carry UltraHonk (~10s of KB) or WHIR-class (~14 KB) proofs — those go to EFSBytes with a REF, which already works. No new surface; stated so nobody discovers it in Phase 0.

### 2.2 Trusted setup vs credible neutrality (C-2)

The question: may a ceremony-dependent SNARK sit inside a 100-year credibly-neutral archive standard?

**RULING:** three tiers.
1. **Kernel/admission tier: moot by 2.1** — no proofs at admission, so no ceremony can ever be load-bearing for record validity or authorship. This is the important guarantee, and it is structural, not policy.
2. **Codex-blessed convention tier** (circuits the EFS spec itself names — e.g. a standard membership circuit for anonymous advisories): **transparent-or-universal setups preferred; per-circuit ceremonies acceptable only with a published, independently verifiable transcript and a stated replacement path.** Rationale: a per-circuit Groth16 ceremony is a 1-of-N honesty assumption (Semaphore's 2024 ceremony had 400+ participants — one honest participant suffices; VERIFIED docs claim); that is *weaker than keccak* but *auditable*, and for optional sidecar features it does not touch neutrality of the archive itself. What credible neutrality cannot absorb is an *unreplaceable* ceremony artifact — hence every blessed circuit is a **versioned convention row** (`zk.circuit.<name>.v1`, v2 mintable later), never Etched. The 100-year horizon argues for transparent systems as they mature: PSE's WHIR-based client-side-proving initiative is explicitly targeting post-quantum, transparent, sub-1M-gas L1 verification (mid-2026, ongoing) — by the time EFS blesses anything, the transparent option should be affordable; prefer it.
3. **Permissionless tier** (anyone's circuits under their own key-TAGDEFs): not EFS's business — enumeration ≠ endorsement, lenses grade the attester as always.

**The PQ asymmetry (100-year honesty, and it cuts both ways):** Groth16/KZG soundness rests on pairing-group hardness — **a CRQC forges proofs**, so a proof-carried record's *authority* degrades at the same epoch E that secp256k1 signatures do; anchored proofs degrade to "existed-before-epoch-E," exactly parallel to identity.md's posture, and nullifier-based uniqueness ("one signal per member") becomes forgeable post-CRQC. But Groth16 is *perfectly zero-knowledge* (statistical simulation — PLAUSIBLE, standard result of Groth 2016), so **the anonymity of past proofs does NOT retroactively break** — HNDL harvesting does not deanonymize provers. Net: ZK-for-privacy is HNDL-safe; ZK-for-authority is epoch-bounded like everything else. The read layer's existing epoch-grading absorbs this with zero new machinery; the WHIR/STARK track removes even the soundness cliff when adopted.

### 2.3 Nullifiers vs EFS revocation — and the admission-confluence check (C-3)

Shapes compared: EFS revocation is a **monotone grow-only G-set** consumed at *read* time; nullifier sets are also grow-only. The shapes are compatible; the *placement* is everything.

- Presence-facts ("nullifier N was used") are monotone and replicate cleanly — they may be recorded, evidenced, unioned.
- But **consuming** a nullifier ("reject the second use") is a *negative existence check* — non-monotone: two venues receiving the two uses in different orders would admit disjoint sets forever. This is byte-for-byte the disease the master invariant exists to kill (the dup-gate/cap-gate class the envelope pass deleted three times).
- **RULING N-1 (the frozen sentence, §4-R1): no kernel admission check may read nullifier, commitment, or any proof-derived state; ZK state never gates admission.** Where does double-signal prevention live, then? Two legal homes, both already shaped in v2: (i) **sibling contracts** (use-case d): the nullifier mapping is venue-local contract state, honestly chain-bound like every gate — a proof replayed to another chain's sibling *will* signal again there, so circuit **scope must bind the verifier's chain/address** when once-per-member matters (state this in the convention text; it is the one real tension between chain-free records and nullifier semantics); (ii) **the read layer** (use-cases b/c): duplicate nullifiers across portable records are **duplicity-shaped evidence** — the resolver treats "two records, same nullifier, same scope" the way it treats SeqCollision: never silently merged, never an admission failure, surfaced as evidence with both records rendered. This maps ZK reuse onto the existing EQUIVOCAL/evidence machinery with a Durable, post-freeze-addable flag (the read-grade flag family is closed-set-revisable — verified read-lens-spec §2.4 / attack-privacy F8 adjudication). No hazard remains once N-1 is law.

### 2.4 The broad-no, definitively (C-4)

**Why shielded-pool-for-files fights the mission — the honest paragraph:**

A shielded pool (Zcash Orchard, Aztec's private state model) makes the *existence and linkage* of state private: notes are commitments, spends reveal only nullifiers, the graph lives encrypted client-side and is reconstructed by scanning with viewing keys. Apply that to a filesystem and every mission end breaks at once. **Verify-don't-trust dies:** "anyone verifies path → file → bytes without a trusted indexer" requires that the *path structure itself* be publicly recomputable; in a shielded design the path exists only inside someone's viewing-key scan — verification becomes a capability, which is trust with better cryptography. **Hyperlinkability dies:** a link is a public Schelling name that strangers can dereference forever; a shielded node is by construction dereferenceable only by key-holders — `web3://efs.eth/alice/thesis.pdf` cannot both be a public 100-year hyperlink and a hidden edge. **Composability dies:** contracts read EFS by point-reads on public state (the onchain-completeness line: keyed indexes, bounded reads); encrypted notes are unreadable by contracts without FHE/co-processor machinery the completeness audit already prices as fantasy for v2. **The archive dies more slowly:** a shielded archive's usable lifetime is bounded by viewing-key custody and scan infrastructure — lose the key, and the bytes are cryptographically present and humanly gone; Zcash's own ecosystem is re-architecting scanning (ECC's Tachyon work targets the sync-cost wall — PLAUSIBLE, not re-verified this round) precisely because "scan everything with a key" degrades as history grows, and a 100-year archive is nothing but grown history. EFS's residual metadata leakage is not a defect to be engineered away by a pool; it is **the purchase price of a public, verifiable, linkable, composable archive**, and the design already says so (privacy.md §1, §8). EFS must never market otherwise.

**The escape hatch, named (what a user who truly needs a hidden graph does):**
1. **The vault pattern (zero new surface, works at year-0):** put the *entire private subtree inside one ciphertext*. Serialize the folder tree + files into a single archive blob (a CAR/tar-like bundle), encrypt client-side, store via EFSBytes under one DATA object with `contentEncryption`, placed (if at all) under a salted TAGDEF. The graph *inside* is invisible — EFS sees one opaque object of bucketed size. Costs: no per-file addressability, no partial sharing, re-upload on change (or chunk-level diffing inside the ciphertext — client concern). This is the honest maximum: **graph privacy by not putting the graph on EFS**, expressible today with rows already reserved (contentEncryption E6, salted family D3/D4, keyWrap E5, padding conventions §H).
2. **An actual privacy chain for the hot state, EFS for the commitment:** keep the private working set on Aztec (mainnet phased launch: Ignition consensus live 2025-11, user transactions early 2026 — [Aztec's own roadmap](https://aztec.network/blog/road-to-mainnet) and [launch coverage](https://www.coindesk.com/markets/2025/11/20/privacy-focused-aztec-network-s-ignition-chain-lights-up-on-ethereum); note a critical vulnerability was disclosed 2026-03 mid-rollout — young stack), and anchor only content commitments into EFS for permanence. EFS's job is the permanent public spine; renting a shielded pool from a chain purpose-built for it is architecture, not failure.
3. What EFS will **not** do: grow a shielded tier of its own. Recorded so silence doesn't decide.

### 2.5 The author-word resolution (the lens tension in (b)/(c), ruled)

First-attester-wins needs an author word; the envelope demands `recovered == author` and v1 admission requires **address-shaped** authors (envelope adopted core; identity ID-SHAPE taxonomy: digest-shaped words are **reserved and rejected** with `ReservedAuthorShape` — verified identity.md). So a year-0 record *cannot* be authored by "a group." The resolution is a two-era pattern:

- **Year-0 (ships with zero new surface): the group-publisher pattern.** A group runs an ordinary EOA (org-key doctrine already exists — identity.md org-as-lens-list). Members submit Semaphore-class proofs; the publisher key wraps each into an ordinary claim whose **body carries the proof**; the publisher's charter (a published LIST/DATA) states "this key publishes only claims carrying valid proofs for group root R." Lenses subscribe to the publisher word like any author. **Authority comes from the embedded proof, not the key** — any reader re-verifies the proof client-side (P-A), so a dishonest publisher cannot *forge* membership claims (it can only censor, and censorship-resistance = anyone-can-be-a-publisher, since the proof, not the key, is the authority; two publishers of the same group are two lens entries). First-attester-wins works unmodified: the author word is the publisher; per-member dedup is nullifier-evidence at the read layer (§2.3). This is the sealed-sender shape: transport identity ≠ asserted identity.
- **KEL-era (post-freeze-addable, already reserved): native group authorship.** A digest-shaped **group identity word** (e.g. derived from the group commitment root under a then-minted domain constant) becomes a legal author via a successor/peer kernel that admits a proof-carrying scheme, exactly the mechanism already reserved for KEL/P-256/PQ (peer kernel + frozen union-read; additive version-skew is legal per envelope amendment 9). Year-0 kernels rejecting those records is the *designed* behavior of the reserved digest space — no pollution, no squatting, full additivity. Sufficiency is tested in §4.

### 2.6 The coupling rule extended (G9-parallel)

Identity.md's one catastrophic non-monotone coupling: key-wrap targets must be independent of the identity signing key. The ZK analogue, stated as doctrine now (convention text, not kernel): **group-credential secrets (Semaphore-class identity scalars) MUST be independent of the identity signing key and of wrap keys.** The common shortcut — derive the ZK identity by signing a fixed message and hashing the signature — welds the domains: EOA theft would then hand the thief the victim's *anonymous* memberships (impersonation inside every group, deanonymization leverage via nullifier correlation), and deterministic derivation makes rotation impossible without leaving every group. Credential keys are a third key domain with the same independence rule and the same per-persona guidance (distinct credential per persona, or the commitment links them — the V3 encryptionKey lesson applied forward).

| b | Anonymous attestation into a lens | **ADOPT-CONVENTION (group-publisher, year-0) + WATCH (native group authorship, KEL-era)** | nothing new (digest author space + peer-kernel path already reserved); rides §4-R1 | author word = publisher key; authority = proof in body; dedup = nullifier evidence at read layer |

| f | SSE / encrypted tag search | **ADOPT-CONVENTION (personal blind-index); REJECT (cross-user encrypted search)** | nothing | single-user SSE is safe and cheap; multi-user SSE is the leakage-abuse literature's kill zone |

### (a) Private membership for keyWrap recipients — does ZK supersede, complement, or duplicate random occurrence keys?

**Answer: complement — and for EFS's own threat model, mostly duplicate.** Decompose what the Fileverse-class problem needs: (1) hide *who can read F* from the public; (2) hide it from *other recipients*; (3) resist brute-force over candidate identities; (4) let recipients find their wrap.

The standing ruling (E5/C6 + attack-privacy A1) already delivers (1) and (2): occurrence keys are **random** in the private tier, so the wrap slot encodes nothing about the recipient; wraps carry no key-id hints; recipients trial-decrypt (O(wraps)); the granter keeps a local recipient→key map for re-key/revoke. For (3): EFS wraps target **published high-entropy public keys** (`encryptionKey` row), not emails — there is no low-entropy identifier to dictionary-attack, which is the *entire* problem Fileverse's vOPRF server exists to solve. One additional fact worth pinning in convention text: the wrap ciphertexts themselves must be key-private — X25519/HPKE without key-id and ML-KEM are anonymity-preserving (ANO-CCA for Kyber-class KEMs — PLAUSIBLE, literature-recalled: Xagawa et al. line of work), so the AEAD/KEM suite chosen for the wrap format MUST come from the key-private subset and the KEM registry text should say so (one sentence added to the C3/E5 conventions; not a new row).

What would ZK *add*? A Semaphore-style membership commitment for "recipients of F" lets someone **prove** they're a recipient without decrypting — but proving-you-can-read is use-case (d) (a gate question), not a confidentiality question; and publishing a per-file recipient-set commitment is itself a linkable artifact (its updates leak re-key timing — the same "permission existence and update timing" leak Fileverse admits). Adding it *by default* would leak more, not less. **Ruling: ZK membership does not supersede random occurrence keys; it composes with them only when a gate needs provable readership (→ d).**

The OPRF pattern earns a narrower home: **the OS tier's "share to an email/name" flow** (no published encryptionKey exists for an email). There, an optional, self-hostable, threshold-able OPRF evaluator (vOPRF-ID direction) hardening low-entropy identifiers is worth an SDK convention — with the honest label that it reintroduces a liveness server into exactly one flow, must never be on the walk-away path for the user's *own* data, and is denied any role in the protocol. **Zero freeze surface** (all off-chain; the permission JSON is just bytes in EFSBytes/DATA).

### (b) Anonymous attestation into a lens

Target capability: "some member of trusted set S asserts X" — anonymous deny-advisories (a security researcher flags a package without painting a target on themselves; a moderation collective labels content without individual attribution; whistleblower curation).

**Ruled shape (year-0, ADOPT-CONVENTION):** the §2.5 group-publisher pattern. Mechanics, precisely: group root R published as an ordinary VAL claim under the group's container (a user key-TAGDEF, e.g. `<group>/zk/root` — permissionless, no reserved row); member generates a Semaphore-class proof with **message = the claim's recordDigest (binding proof→record so it can't be re-attached)** and **scope = keccak(scopeDomain, definitionId, epoch)** (a convention constant, not a kernel constant); publisher submits the claim with proof bytes in the VAL tail (Groth16 fits; Honk goes EFSBytes). Readers verify: signature → publisher; proof → membership in R *as of the root the proof names* (roots are updatable; the proof pins which). Deny-advisory consumers get exactly what §3.4 of read-lens-spec needs — the advisory is an ordinary graded TAG by the publisher word, point-readable — plus a client-checkable membership upgrade.
**Honest limits, named:** (i) the anonymity set is the *group at root R*, and root-update history is public — a group of 3 is barely anonymous, and member-add timing correlates (the Fileverse leak class, unchanged by ZK); (ii) the publisher sees submission timing/IP unless members submit through mixnet/OHTTP transport (out of protocol; SDK documents it); (iii) revoking an anonymous advisory: only the *publisher* can REVOKE (it authored the claim) — a member who wants their anonymous assertion withdrawn must ask the publisher or the group rotates the root; state this in the convention (revocation-authority follows the author word, always). (iv) Lens semantics unchanged: the lens trusts *the group as curated by its root-maintainer* — root governance is the real trust decision and must be disclosed like a lens (LC1-parallel).
**KEL-era (WATCH):** native digest-shaped group authors per §2.5, trigger = KEL design round.

### (c) Proof-of-authorship-without-identity

"An author in set S wrote this record" — the stealth lane owns unlinkable authorship; this lane states the composition interface and does not duplicate it:

- Stealth lane provides the **author word**: a fresh address-shaped one-time key — admissible year-0, no envelope change.
- ZK provides the **credential**: proof in the record body that the one-time author key is controlled by / derived from a member of committed set S. Two circuit families: (i) secp256k1-native (spartan-ecdsa-class, ~8k constraints, browser-provable, **unaudited**; needs member pubkeys, which are only recoverable for addresses that have transacted — real onboarding constraint); (ii) Semaphore-class over separate credential keys (audited, ceremony-done, but imports the §2.6 third-key-domain burden and the commitment-registration step). Binding: message = recordDigest **and** the circuit must additionally bind the *author word itself* (prove control of the stealth key inside the circuit, or sign-over-proof) — otherwise a valid proof could be lifted onto a different author's re-signed record; this bind-the-carrier requirement is the one non-obvious spec point, stated here so the convention gets it right.
- Lens consumption: the record's author is the stealth word (position resolution unchanged); the *grade upgrade* "member-of-S" is a read-layer flag computed from the proof — Durable, post-freeze-addable (flag family revisable).
- **Verdict WATCH:** nothing freeze-sensitive; blocked on the stealth lane's derivation reservation and on circuit auditing maturity, not on EFS surface. Trigger to revisit: stealth lane lands + one audited secp256k1-membership circuit exists.

### (d) Private lens-membership for GATE/contract reads

First, the honest reframe: **a public chain cannot enforce "may read" — reads are unobservable and un-gateable** (privacy.md §1: no read-ACL, ever). What a contract can gate is an *action* (mint, unlock, vote, admit, pay). So the real capability: **membership-gated on-chain actions without a public member list** — and this is exactly production Semaphore: group root on-chain, `validateProof(root, nullifier, message, scope, proof)`, nullifier mapping prevents re-use.

**Ruled shape (ADOPT-CONVENTION):** the EFS-native wiring is what's new and it needs nothing frozen — the **group root lives as an ordinary EFS claim** (PIN VAL under the group's key-TAGDEF, cardinality-1, superseding on rotation), and the sibling verifier reads it via the Etched `getSlot`/`getClaim` point reads, then verifies Groth16 (~200–350k gas — the gate's cost, not the kernel's) and consumes the nullifier locally. Root freshness is the standard gate problem, already solved by the read spec: the gate applies its checkpoint-age/expiry policy to the root claim like any safety-class key (read-lens-spec §5.4/§9.C shapes). Membership *changes* are root supersessions — LWW slot semantics give the gate "current root" for free. What this buys over a plaintext on-chain allowlist: the list is never public, removals don't leak who was removed (only that the root moved), and the member's *action* is unlinkable to their identity. What it does not buy: hiding that gating exists, group size class (root-update cadence leaks activity), or anything against a malicious group admin (who defines the tree).
Deliverable when built: one reference sibling (`EFSSemaphoreGate`) + the root-claim key convention. **Zero kernel surface; zero reserved rows** — verified against the closed reserved-key table by the §4 sufficiency test.

### (e) Selective disclosure — prove a property of an encrypted record without revealing it

Examples: prove a claim was made before date D without opening it; prove encrypted file F contains chunk with hash H (DMCA-counter / plagiarism proofs); prove an encrypted VAL parses to `age ≥ 18`-class predicates; prove two ciphertexts encrypt the same plaintext (dedup arbitration without disclosure).

Feasibility mid-2026: general-purpose circuits over hash commitments are routine; costs hinge on the hash. EFS already made the lucky choice: **EFSBytes commits per-chunk SHA-256 (row C4)** — SHA-256 is several-fold cheaper in-circuit than keccak (order 25–30k vs ~150k R1CS constraints per block — PLAUSIBLE, standard figures), so chunk-inclusion proofs over EFS large files are the cheap case; keccak-committed record digests are the expensive case (still feasible, just heavier, and zkVMs absorb it for non-interactive uses). AEAD-ciphertext-to-plaintext bindings need encryption-in-circuit or committing AEAD constructions — the convention should note that a **commit-then-encrypt discipline** (commit to plaintext, encrypt, prove against the commitment) is the SNARK-friendly private-tier layout; that is an SDK/format guidance line inside the existing `contentEncryption` conventions, not a row.
**Verdict WATCH:** proof-in-body (P-A) carries it; sizes fit (Groth16 inline; larger → EFSBytes); nothing frozen. Trigger: first concrete app demand (the coercion/journalism lane and marketplace lane are the likely callers).

### (f) Structured encryption / SSE for encrypted tag search — mini-ruling

Question: can lens/tag queries run over encrypted metadata without ZK/FHE hammers?

**Split by trust topology, which is what the literature actually keys on:**
- **Single-user (my own private tags, searched by my own devices): YES — ADOPT-CONVENTION.** The construction is a *blind index*: tag values HMAC'd under a user-held key (HKDF from the salted-family secret — the D3 HKDF salts already give device-loss recovery), published as ordinary VAL claims. My devices recompute the HMAC and point-read; nobody else learns the vocabulary. Leakage, named honestly: slot-key *frequency and co-occurrence* remain visible (how many things carry tag X, which objects cluster) — this deliberately trades the L9 randomize-sensitive-VALs rule for searchability, per-user-salted so there is no cross-user equality oracle; timing/count leakage is the price and the padding/dummy-claim conventions apply. Zero freeze surface (user key-TAGDEFs + VAL claims).
- **Multi-user / shared encrypted index queried by an untrusted party: NO — REJECT for EFS.** This is classical SSE, and its leakage profiles (access pattern + search pattern + volume) are broken in practice by two decades of attacks: IKK co-occurrence recovery ([Islam–Kuzu–Kantarcioglu, NDSS 2012 — via the leakage-abuse literature](https://eprint.iacr.org/2016/718.pdf)), the count/leakage-abuse attacks of [Cash–Grubbs–Perry–Ristenpart (CCS 2015 / ePrint 2016/718)](https://eprint.iacr.org/2016/718.pdf), **file-injection** attacks with 100% query recovery ([Zhang–Katz–Papamanthou, USENIX 2016; refinements ongoing](https://link.springer.com/chapter/10.1007/978-3-031-70896-1_21)) — and EFS is *permissionless-write*, i.e. an adversary can inject records into your index by design, which is the file-injection attack's ideal habitat; re-evaluations show attacks remain potent under realistic priors ([Blackstone–Kamara–Moataz, "Revisiting Leakage Abuse Attacks," NDSS 2020 / ePrint 2019/1175](https://eprint.iacr.org/2019/1175)), with volume-only attacks (Kellaris et al. 2016 — PLAUSIBLE cite) and update-leakage attacks ([CCS 2023 line](https://dl.acm.org/doi/10.1145/3576915.3623085)) closing the remaining gaps. On a *permanent* chain every leaked access pattern is harvestable forever — SSE's leakage is amortized-catastrophic here. Forward/backward-private dynamic SSE mitigates but does not close the class, and requires an interactive server EFS refuses.
- **FHE:** stays REJECT-for-v2 per privacy.md §6 (cost model vs on-chain composability); nothing this round moves it.
**Net:** lens/tag queries over encrypted metadata exist for the *owner* (blind index, cheap, now) and do not exist safely for *shared* indexes; shared private search waits for a fundamentally different tool (PIR-class, the read-path lane's frontier), not for a cleverer SSE.

## 4. Freeze-sensitive reservations (PART D — the loud section)

**Headline: NOTHING new must be Etched for ZK — and here is the adversarial check.** Reserving is cheap and omissions are permanent, so each candidate was tested by enumerating what actually shipping the feature later requires and checking each requirement against the already-reserved surface. One frozen *sentence* is demanded (R1); every other candidate is CONVENTION or REJECT, with the sufficiency argument shown so the ceremony can audit it.

**R1 — ROW-CLASS (Codex normative text, zero state): the ZK-inertness rule.**
> *"No kernel admission check may read nullifier sets, membership commitments, proof-verification results, or any ZK-derived state; proof verification is read-layer or sibling-contract, never admission. Duplicate-nullifier facts are evidence, never rejections."*
Classification: this is a **frozen doctrine sentence** in the same class as the master admission invariant (which it instantiates) and the key-wrap independence rule — it must be written into the Etched Codex text so no Phase-0 implementer wires a verifier into `submit`. It reserves no bytes. Sufficiency: trivially total — it is a prohibition. **ADOPT; ceremony copyedit only.**

**R2 — reserved genesis row(s) for ZK (`zk.groupRoot` / `zk.identityCommitment` / `zk.nullifier`): REJECT.**
Sufficiency test for shipping (b)/(d) later without them: group roots and identity commitments are ordinary VAL claims under **permissionless user key-TAGDEFs** — no reserved-row carve-out needed because (i) they are read by *derived point lookup by parties who know the convention* (the gate/verifier), never required to be walk-enumerable by strangers; (ii) the correctness-failure argument that earned `encryptionKey` its row (silent mis-encryption) has no analogue — a wrong root/commitment fails *loudly* (proof rejects); (iii) nullifiers never belong on EFS at all (sibling state or read-layer evidence, per R1). Junk-reservation risk is real (a frozen row shape would have to guess circuit family, root arity, epoch discipline — exactly the "unwritten code cannot be hash-pinned" trap). **Convention rows in the registry (§H class) when the first real group ships; nothing minted at genesis.**

**R3 — a reserved envelope signature-scheme byte for proof-carrying records (e.g. 0x04 = "group proof"): REJECT.**
Sufficiency test for KEL-era native group authorship (§2.5): requires (1) a legal author-word shape → **already reserved**: digest-shaped words are reserved-and-rejected (`ReservedAuthorShape`), so the space stays unpolluted and successor kernels may assign it (verified identity.md frozen core); (2) an admission path for proof-verified records → **already reserved as a mechanism**: peer/successor kernel + frozen union-read + additive version-skew legality (envelope amendment 9; the KEL precedent), and the algoTag family shows the pattern of *deliberately deferred minting* until the scheme is real (identity.md F1: the PQ tag is minted only when NIST-final + verifier exist); (3) read semantics → Durable (closed grade set is spec-revisable; flags batch with P3). Freezing a layout *now* for an unbuilt proof system is the same mistake as pre-wiring the KEL registry address — the design already rejected that class. The one cost of not reserving: year-0 kernels will intrinsic-reject proof-authored envelopes forever (chain-local, recoverable by peer kernel exactly as KEL records are) — accepted, it is the designed succession path.

**R4 — a `DOMAIN_ZKGROUP_V1` derivation-domain constant: REJECT (additive later).**
Domain constants are spec-owned printable preimages; new `v1`-suffixed domains are additive by the hash-migration/new-domain rule (deterministic-ids §1/§13.6). A group-word derivation formula belongs to the (future) group-authorship spec and mints its constant then. The only non-additive derivation surface is the **registry resolver-gate set** — which matters only if group identities must become first-class *registered objects* (walkable containers, `getObject`-classifiable). Checked: they don't — OPAQUE targets are legal in user LIST charters and TAG/PIN payloads (closed targetKind set includes TARGETKIND_OPAQUE; the OPAQUE-forbidden rule binds *reserved* rows only — verified codex-kinds K5), so lenses, wraps, and edges can reference group words opaquely at year-0, and first-class-ness can ride a successor kernel if ever wanted. **If James disagrees and wants first-class group objects on the year-0 kernel, that — and only that — becomes a now-or-never resolver-gate reservation; flagged as his call in §6.**

**R5 — key-privacy sentence in the KEM registry text (rides existing rows): ADOPT (one sentence, ceremony copyedit).**
The C3 `encryptionKey` KEM/KEX registry and E5 keyWrap conventions gain: *"registry KEM/AEAD suites for private-tier wraps MUST be key-private (anonymity-preserving; no key-id hints in wrap blobs)"* — without it, a non-key-private suite silently rebuilds the recipient oracle that A1 killed, one layer down. This amends reserved-row *text*, not shape; it is freeze-sensitive only because the row text freezes with the table. (Cross-check: attack-privacy F12 already flagged the blob-hint half; this closes the suite-choice half.)

**R6 — credential-key independence rule (G9-extension, §2.6): CONVENTION (identity doc text), not kernel.** Belongs beside the ENCRYPTION row of the consequence table; no frozen surface, but should land in the same doc revision that carries the other privacy-pass amendments so it isn't lost.

**R7 — proof-size facts (VAL ≤ 8192 vs proof families): no reservation, one spec note.** Groth16-class fits VAL; Honk/WHIR/STARK-class rides EFSBytes REF. Both paths exist; record the note in the conventions so nobody proposes widening VAL for proofs (REJECT any such widening — it's envelope surface).

**Interaction check against the existing reserved privacy surface (fs-pass-freeze-reservations):** R1–R7 touch none of the pinned shapes — keyWrap stays TAG-only with random occurrence keys (ZK ruled complementary, §3a); salted-TAGDEF family unaffected (blind-index rides its HKDF salts); `claimedAt=0` rider and padding conventions compose with proof-carrying records unchanged; C4's SHA-256 chunk words are *helped* (cheaper circuits, §3e). No amendment to any D/E-row shape is demanded by this lane.

## 6. Decisions for James

Plain English, with examples. Recommendations attached.

**D1 — Ratify the line: ZK as sidecar evidence, never as kernel behavior.**
What it means: people will be able to attach cryptographic proofs to EFS records ("someone in this security team flagged this file" — without which member; "the caller is on the allowlist" — without the allowlist being public), and contracts can check those proofs in ordinary redeployable contracts. But the frozen kernel will never check a proof to decide whether a record is admitted, and EFS will never hide the file graph itself.
Options: (1) **Adopt as written** — narrow-yes/broad-no per this file. (2) Broaden — also reserve first-class on-kernel group identities now (costs a resolver-gate reservation, R4's flagged variant; buys walkable group containers at year-0). (3) Narrow — reject even the conventions, ZK stays entirely out of EFS docs.
**Recommendation: (1).** Option 2 reserves shape for an unbuilt spec (the trap the KEL decision already avoided); option 3 forfeits real, cheap capabilities (anonymous advisories are the deny-layer's missing piece for whistleblower-grade sources).

**D2 — The one frozen sentence (R1).** "ZK state never gates admission." This is the same move as "no admission check reads revocable state," extended to proofs — it keeps every future ZK feature un-dangerous to replication.
Options: adopt / don't.
**Recommendation: adopt — it is one sentence, and it is the fence that makes everything else safe.**

**D3 — Circuit-blessing policy for the Codex (trusted-setup posture).**
Example: when EFS blesses a standard "anonymous advisory" circuit, may that circuit require a trusted-setup ceremony (like Semaphore's 2024 ceremony, 400+ participants, one-honest-participant assumption), or must it be a transparent system (no ceremony, also post-quantum, currently ~2–6× the verify gas)?
Options: (1) **Transparent-preferred, ceremony-tolerated** — Groth16-class allowed for optional features if the ceremony transcript is published and the circuit row is versioned/replaceable; migrate to transparent as WHIR-class verifiers mature. (2) Transparent-only from day one — cleaner story, pays gas + waits on maturing tooling. (3) No policy — decide per feature.
**Recommendation: (1)**, with the R1 fence making even a bad ceremony non-catastrophic (worst case: an optional sidecar feature is discredited and its convention row is superseded — the archive itself is untouched).

**D4 — Bless the year-0 "group publisher" convention?**
Example: OSV-like security collectives publishing anonymous deny-advisories into lenses this year, using Semaphore v4 as-is, with proofs anyone can re-check. It ships with zero kernel changes.
Options: (1) **Bless it** (write the convention: root-claim key shape, scope derivation, proof-in-body binding, nullifier-as-evidence, revocation-authority note). (2) Wait for a concrete user to ask.
**Recommendation: (1) write the convention now, build nothing yet** — conventions are how EFS keeps later implementations compatible, and this one costs a page.

**D5 — (flagged from R4) Do you want group identities as first-class registered objects on the year-0 kernel?**
