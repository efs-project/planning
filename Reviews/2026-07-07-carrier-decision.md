# 2026-07-07 — Carrier decision (EAS vs native kernel vs alternatives)

**Context:** After the v2 decisions (deterministic IDs, tag-core, string-only props, chain-free EIP-712 Merkle envelope for portability), EFS uses ~none of EAS's semantics and the ecosystem-legibility that justified staying on EAS *inverted* (a standard-looking attestation with 4 dead fields + a redefined attester lies to every EAS tool). James asked to resolve the carrier before the Fable pass: what replaces EAS, is it verifiable/easy, can we actually do it, does it help portability without extra clicks, are there good EVM alternatives. 4-agent workflow `wq392r0mi` (one explorer glitched → placeholder output; Q2/Q5 covered by the other explorers + corpus + synthesizer with source citations — the dedicated EAS-standards web pass was thinner than intended; "EAS is not a ratified EIP" verified from per-chain address/version divergence rather than a standards search).

## Ruling: native envelope kernel as canonical carrier + optional EAS-attestation export

**Conditional on the portability gate (below).** Attacking the ruling first (honest pro-EAS steelman): EAS is audited custody of auth/store/revoke; easscan legibility + the "attestation" neutrality umbrella are real goodwill; EAS delegation *already* gives gasless relay with author-as-attester today; the chain-free signature/identity spec is novel and Etched (the forSchema-imitation risk). Native wins anyway on two reasons that survive:

1. **The scary novelty is unavoidable, so it's not a cost of leaving EAS.** James already decided the chain-free envelope. Every EAS signature domain binds `{chainId, verifyingContract, per-chain version}` (verified: `eas-sdk` offchain.ts, `EIP1271Verifier.sol`) — a chain-free signature *cannot* be an EAS signature. So staying on EAS doesn't buy back the hard spec work; it wraps a lying carrier around a signature EAS can't validate, *without* buying portability. The real choice is "does the canonical record also pretend to be an EAS attestation (dead fields + lying attester = legibility inversion) or stop pretending?" Full-EAS adds the lie without the mission property. Collapses the main pro-EAS argument.
2. **Code cost is negative; the multiAttest dependency is trivial.** Verified `EAS.sol:402-486`: the atomicity "engine" is a for-loop (store each, then loop onAttest, revert-all-on-false) — ~100–200 LoC to reimplement, *stronger* natively (atomicity by construction, parents-first kernel-enforced, no bytecode pin, no per-chain conformance suite, no mid-batch `_db` footgun). Net new kernel ≈ 500–900 LoC replacing ~1,205 LoC of EAS machinery; the ~2,900 LoC validation is already bespoke/EAS-unaudited, ports at ~10–20% churn. Not giving up an audited wheel — giving up an audited *hub that no longer turns the axle*.

The **export** neutralizes the one real loss (legibility): a separate non-Etched `EASExporter` mints a real EAS attestation on demand, attester = exporter, honestly labeled a derived mirror — EAS as opt-in tool, not lying carrier.

## Answers to the five questions

- **Q1 replacement:** native kernel — chain-free EIP-712 Merkle-signed envelope, signature = auth, recovered signer = author. `{author, seq (TID clock), prev, recordsRoot, count}` + `Record[] {op, kindTag, body}`; sign the root once; anyone submits. Verifiable via `ecrecover` (offline, year-100, no server). Easy for writers; honest DX loss = no easscan (export recovers it).
- **Q2 reinvent:** EAS traction is real (Coinbase Verifications, Gitcoin Passport, OP predeploys) — mostly optics. **EAS is NOT a ratified EIP** — one vendor's suite, per-chain divergent addresses + version strings. The standard is EIP-712/ecrecover, which native uses more natively. Mechanics aren't hubris (smaller, deletes footguns incl. timestamp-in-UID); the novel signature/replay/identity *composition* is the real risk → external-review gate.
- **Q3 feasibility:** yes, conditionally. Burden = verification, not headcount (invariant suite + external review + executable Codex). Fails early (yr 1–5) or never. +2–4 weeks + disproportionate verification; ship bare-EOA first-class (KEL opt-in). Keep abort-to-EAS fallback real until the gate passes.
- **Q4 portability/clicks (decisive):** native delivers 1-sig portable writes by construction; EAS cannot (delegation = chain-bound + N sigs + sequential nonces; envelope-relayer = lying attester). Native: one sig over the DAG, recovered signer = author, `msg.sender` never in the auth path — relay/sponsor/replicate all fall out of one mechanism. Portable revocation is a bonus native has and EAS lacks (chain-local).
- **Q5 alternatives:** none fit better. Verax = counter IDs (anti-LOCKSS, privileged massImport migration); Sign Protocol = centralized SignScan indexer; VC/SD-JWT = composability + wallet-production fails; did:ion walked back to did:web. The one lineage worth importing (KERI/did:webvh key-log) is already absorbed in the identity design.

## The gate — RESOLVED (James, 2026-07-07): portability is first-class → NATIVE

James ruled cross-chain portability a first-class requirement (hundreds of L2s/L3s; one-attester-per-chain is futile; write-once-copy-forward-read-natively with no author work and no trust in the copier — the Vitalik-config case). So native is the carrier. The junk-fields EAS-carrier middle is dead; the stay-on-EAS-with-delegation fallback remains only as the pre-freeze abort ramp if external review fails.

## Revocation — free to propagate, best-effort to complete (the one caveat)

A revoke is just another signed record: free (one author signature; propagation paid by whoever copies), copy-pasteable, verifiable, and *portable* — strictly better than EAS (whose revocation is chain-local + msg.sender-keyed and doesn't travel). So propagation has no gotcha. The gotcha is on the READ side and is inherent: **presence is monotone (copying adds truth), revocation is about absence** — to trust "R is not revoked" you must know you aren't missing a `REVOKE(R)`, and a withheld/not-yet-copied revoke makes a foreign chain serve revoked data as valid with the author's real signature (e.g. a security-bugged `config-v1` that was revoked, copied to an L3 without its revoke). So "not revoked" cross-chain = "not as far as this chain knows," never certain. Confined to cross-chain reads (home chain is certain, one lookup); mitigated by author-set **expiry** (the TLS-cert answer), signed freshness heads, or pull-latest-before-trust; never fully closed without a consensus substrate (the "portable currency" we chose not to build). **Practical rule: apps use expiry for anything where serving stale data is dangerous.**

## Decision rule + experiment

Flip to stay-on-EAS if: (1) portability turns out not mission-critical now (biggest flip — pressure-test before freeze); (2) the envelope+identity spec fails independent external review; (3) the verification bill blows the freeze window. Flip to pure-native (drop the export) if no consumer wants easscan. Does NOT flip it: gas overhead (~5–10%, second-order), easscan loss (export recovers), neutrality optic (already punctured by EAS per-chain divergence), maintenance-headcount fear (burden is verification, code shrinks).

**Smallest experiment (days, devnet):** an "envelope conformance harness" — minimal `submit()` with a chain-free domain; prove three properties that must hold *simultaneously* and have no off-the-shelf answer: (a) **portable** — same signed envelope → identical per-author state on two chain-ids; (b) **replay-safe** — resubmit → idempotent no-op; (c) **1-signature UX** — sign an 8-record file DAG with one `eth_signTypedData_v4`, relay from a different account, assert recovered author (not relayer) recorded. Plus one portable-REVOKE replayed onto both chains. If (a)+(b) can't both hold, that's the cheap abort signal — before the freeze.
