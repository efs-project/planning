# Secure prompts, trusted chrome, and anti-spoofing UX — research digest
**Corpus:** 2026-07-07-clientv2-corpus. **Agent lane:** secure-ui. **Date:** 2026-07-07.

## Why this lane is existential for EFS client v2

The EFS handoff commits to a hard architecture: a Shell that owns "prompt chrome apps cannot imitate," Ring-3 apps in SES compartments, and a Kernel that mediates keys/wallet/network. Every one of those guarantees ultimately rests on the user being able to tell *Shell-drawn* pixels from *app-drawn* pixels while both live inside a single browser tab. That is precisely the problem the desktop-OS and browser-security literature spent 20 years failing to solve cleanly. The "line of death" — the visual boundary between trusted chrome and untrusted content — is **already dead in browsers** (Emily Stark, Chrome security, 2022), and inside a browser page EFS has *no browser chrome to inherit at all*. So EFS is trying to re-establish a trusted path one layer deeper than the layer where the trusted path already broke.

Bottom line up front: **you cannot win the pixel-spoofing war with visual design alone.** Every deployed system that tried (SiteKey security images, EV certificates, custom cursors, colored borders drawn by the untrusted party) failed. The systems that worked moved the security-critical decision *out of the spoofable surface entirely* (out-of-process pickers, kernel-intercepted secure attention keys, hypervisor-drawn window borders) or *removed the spoofable secret* (passkeys). EFS's defensible path is a combination of: (1) out-of-app-viewport rendering the Kernel controls, (2) structured/clear transaction decoding instead of blind signing, (3) interaction gating to defeat clickjacking/keyjacking, (4) negative (warning) indicators over positive (trust) indicators, and (5) knowing when to punt to the real hardware-wallet UI. Details below.

## 1. WHAT EXISTS TODAY

### 1.1 The line of death and its death

- **Eric Lawrence, "The Line of Death" (2017-01-14).** The foundational framing: trustworthy UI (browser/OS chrome) must be visually separated from untrusted content, because "if an attacker has full control of a block of pixels, he can make those pixels look like anything he wants, including the UI of the application itself." He enumerates the erosion zones: page title + favicon (fully attacker-controlled), domain (lookalikes), path (falsifiable), content area (fully untrusted). His killer anecdote: a Fortune-500 bank's *entire security team* could not pick the faked PayPal picture-in-picture window out of a screenshot.
- **Emily Stark (Chrome security lead), "The death of the line of death" (2022-12-18).** Argues the line is obsolete: it was never empirically validated, users don't perceive it, it's inconsistent across platforms, and modern browser features (side panels, Payment Handler API, PWA UI) deliberately anchor *web-origin content to browser chrome*, blurring the boundary. Her prescriptions are the load-bearing lesson for EFS: **(a) use negative indicators only** — warn when something is wrong rather than reassure when things are fine, because positive indicators are the thing attackers spoof; **(b) unphishable credentials** (WebAuthn/passkeys) to remove the spoofable secret; **(c) upstream moderation** (email/search/social filtering before the user ever sees the lure).
- **Chromium "Security considerations for browser UI" (living doc).** Codifies the current consensus rules EFS should copy verbatim: "Prefer to warn users when things are amiss, rather than reassure them about neutral or positive security properties"; reuse established UI patterns; never render attacker-controlled content in trusted UI without clear demarcation; prevent occlusion of critical UI; and — critically for interaction gating — remove default-selected accept buttons (anti-keyjacking), require clicks in different locations, and impose **activation delays** (`InputEventActivationProtector`: ~3s for interstitials, ~500ms for smaller surfaces).

### 1.2 Chrome UI spoofing techniques and what browsers did

- **Browser-in-the-Browser (BitB), mr.d0x (2022).** Pure HTML/CSS/JS renders a *fake OS window with a fake address bar* inside the page, typically for fake "Sign in with Google/Microsoft/Apple" SSO popups. "Combine the window design with an iframe... and it's basically indistinguishable." Real Steam-account thefts up to $300k. **No browser fix exists** — it's just page content. The only defenses are behavioral (real SSO popups can be dragged outside the parent window; fakes cannot) and structural (passkeys, which don't route through a spoofable popup). This is the single most important precedent for EFS: *an in-page rectangle that looks like system chrome is trivial and unstoppable by the page itself.*
- **Fullscreen API abuse (feross.org 2016; text/plain 2023).** `Element.requestFullscreen()` wipes all browser chrome; attacker paints a fake browser/OS on the whole screen. Escalated with **Pointer Lock** (hide cursor) + **Keyboard Lock API** (intercept Esc). Browser mitigations: a transient "you are now fullscreen, press Esc to exit" overlay; Chrome requires a **long (~2s) Esc hold** to exit keyboard lock; Chrome added a flag to *prompt before* keyboard lock. Lesson: fullscreen is an attacker's dream and the mitigations are all "make the escape hatch always work and always advertised."
- **Custom-cursor / cursorjacking (Mozilla MFSA-2007-04, MFSA-2015-35; Palant 2021).** CSS `cursor:` with a hotspot offset draws a fake pointer displaced from the real click point, so the user clicks trusted UI while thinking they're clicking elsewhere. Browsers now **block cursor customization that intersects toolbar/chrome regions**. A study found that even disallowing cursor customization still left **16% of participants fooled**, because the attacker can *draw multiple fake cursors in page content*. Lesson: cursor trust is unrecoverable inside a page; interaction gating must not depend on the cursor being where it appears.
- **Document Picture-in-Picture API (shipped Chrome 116, 2023; MDN).** Opens a real always-on-top window populated with arbitrary HTML. Spec deliberately **forbids the origin from setting window position** and constrains size, and the always-on-top window still gets browser-drawn chrome — a direct anti-spoofing design choice. But it's a new always-on-top surface and researchers flag it as a spoofing risk if chrome is ever minimized.
- **Address-bar spoofing** remains a recurring bug class (many CVEs; Invicti roundup). Even the *browser's own* trusted string gets spoofed via race conditions, RTL/Unicode tricks, and `://` parsing bugs (e.g., Chrome download-UI origin-spoofing issue 392818696).

### 1.3 Modal / in-app chrome spoofing (the Web3-specific version)

- **CertiK "Modal Phishing in Web3 Mobile Wallets" (2023).** The wallet's *own* connection/signing modal shows dApp-supplied name, icon, URL, description **without verification**. A malicious dApp impersonates a trusted one; users approve based on the spoofed identity in *trusted wallet chrome*. Mitigation shipped as **WalletConnect Verify API** (attests the real requesting domain). Directly analogous to EFS: if the Shell's prompt shows app-supplied identity metadata, the *app controls the trust signal inside trusted chrome.* The identity in a Shell prompt must be Kernel-derived (package CID / signer / manifest), never app-asserted.

### 1.4 Wallet signing UX failures — the blind-signing catastrophe (2024–2025)

- **Radiant Capital ($50M+, 2024-10-16).** North-Korean UNC4736 malware on ≥3 developer machines showed *legitimate* Gnosis Safe data in the frontend while sending `transferOwnership()` to the Ledgers. Ledgers **don't parse Safe transactions**, so signers blind-signed. Tenderly simulation was run and reviewed — but blind signing meant the review didn't catch it. Lesson: **the display and the signed bytes were different, and the human trusted the display.**
- **Bybit ($1.4–1.5B, 2025-02-21) — largest crypto theft ever.** Lazarus injected malicious JS into the Safe{Wallet} *frontend* via a compromised Safe developer machine (Safe had **no Subresource Integrity, no real-time frontend-integrity alerting**). Signers saw a routine transfer; they actually signed a `delegatecall` that swapped the wallet's implementation. Same root cause as Radiant at 30× scale. Certora/Fireblocks/NCC analyses converge: **multisig is not a control if all signers trust the same spoofable display.**
- **Permit / Permit2 signature phishing.** Off-chain `eth_signTypedData_v4` signatures (gasless, invisible on-chain until used) drove **56.7% of all token-phishing losses in 2024 = ~$494M across 332k victims (+67% YoY)** (ScamSniffer). A single Permit2 signature can drain every token ever approved to the canonical router. EIP-712 typed-data UIs are "notoriously hard to read"; attackers also abuse EIP-712 *normalization* to bypass wallet checks. 2025 losses fell ~83% to ~$84M as clear-signing/scanning matured — evidence the fixes work.
- **Address poisoning.** ~270M attempts / 17M victims / 6,633 confirmed hits / $83.8M (USENIX study, Jul-2022–Jun-2024); a single **$50M** loss on 2024-12-19 from copying a poisoned address. Root cause is **UI truncation** (`0xd9A1…53a91`) — users verify only visible ends while the middle is substituted. A 2025 review of 53 ETH wallets: 16 showed phishing transfers with no warning; only 3 alerted on suspicious addresses.
- **Fake WalletConnect on Google Play ($70k, 2024).** A drainer disguised as a calculator passed Play review, ran 5 months, 10k+ installs. Lesson: **store/distribution review is not a security boundary**; provenance ≠ safety (the EFS handsoff already flags "first-party is provenance, not a security boundary").

### 1.5 What clear-signing / decoding do about it

- **Ledger Clear Signing + ERC-7730 (initiative 2024; EIP-7730 published; v2 in 2026; governance moving to Ethereum Foundation).** A per-contract **descriptor file** enriches ABI/typed-data with formatting + intent so calldata renders as human intent ("Send 1,000 USDC, receive…") instead of hex. Generic Parser (2025) reads ERC-7730 metadata so no per-dApp plugin is needed; v2 adds cross-chain and *software + hardware* wallet support. Trail of Bits (2025-08-27) published "implement EIP-7730 today." This is the emerging standard EFS should track/adopt for its own record/envelope decoding.
- **Rabby / MetaMask+Blockaid.** Rabby simulates every tx and shows **plain-English balance changes + risk scan** pre-approval; MetaMask added Blockaid scanning in 2024. Independent reviews credit simulation+scanning with catching materially more malicious txs. Lesson: **simulate-and-diff ("what will my balances be after?") beats decoding raw fields.** But note simulation depends on an RPC/simulator endpoint — a *network + privacy* dependency EFS's no-ambient-HTTP rule makes non-trivial.
- **Post-Bybit Safe hardening:** byte-by-byte payload validators (pcaversaccio/OpenZeppelin Safe Utils), SRI on the frontend, on-chain Safe Guards that block `setImplementation`/`delegatecall`. Lesson: **verify the hash the hardware shows against an independently computed hash**; don't trust one rendering pipeline.

### 1.6 OS-level secure UI precedents (what worked, what didn't)

- **Secure Attention Key (Ctrl+Alt+Del / SAK; Wikipedia, TCSEC "trusted path").** The *only* consistently successful anti-login-spoofing mechanism. Works because the **kernel intercepts the key combination and suspends all applications** before showing the trusted login — no application can register or fake it. This is the clean model EFS *cannot fully replicate* in a browser: **the web has no secure attention key**, because the page/JS layer can't be guaranteed-suspended by anything the page itself can't also see. (systemd added Ctrl+Alt+Shift+Esc as a Linux SAK in v257, 2024 — the pattern is still alive.)
- **Windows UAC secure desktop (Vista→11).** UAC dims the screen and prompts on an **isolated secure desktop** that normal apps can't draw on or screenshot — architecturally correct. But Vista's over-prompting produced **"UAC fatigue"**: users reflexively clicked Yes. Win7 fixed it by *drastically cutting prompt frequency* (no prompt for signed in-box changes). **Lesson: a correct secure surface is undone by prompting too often.** The secure desktop is the good idea; the prompt budget is the hard part.
- **Qubes OS trusted window decorations (Qubes GUI docs; LWN 2016).** dom0 (trusted) draws **unforgeable colored borders** per security domain; VMs speak only a minimal GUI protocol and can never draw the border or touch dom0's X server. The catch (LWN "colored-border spoofing", 2016): a malicious app can draw *its own fake colored border inside its content* and try to fool users into misreading domain — so even hypervisor-drawn borders need the trusted region to be *outside* any pixels the untrusted party controls, plus user training. Directly relevant: EFS Shell chrome must live in a region the app *cannot paint into*, and even then a fake border drawn in app content is a residual risk.
- **Out-of-process pickers — the real success story.**
  - **iOS `PHPickerViewController` (WWDC20/21).** Runs **out-of-process**; the app cannot access the photo library, cannot read the picker's contents, and **cannot even screenshot the picker**. Only the user-selected item crosses back. Access = the selection itself; no permission prompt needed.
  - **Android Photo Picker / Storage Access Framework.** System-rendered picker; SAF returns a **per-URI capability grant** (read/write/delete flags, persistable across reboot) for exactly what the user picked. Android's embedded photo picker is "rendered in a way that prevents any drawing or overlaying." No broad storage permission required.
  - **Lesson (the strongest positive result in this whole lane):** the picker *is* the grant. Selection = consent = capability, with **no separate spoofable prompt and no ambient permission**. This is the powerbox pattern and it's the single best UX+security primitive EFS should build the Shell around.

### 1.7 Permission-prompt research (the quantitative case against prompts)

- **Felt et al., "How to Ask for Permission" (HotSec 2012).** The canonical taxonomy: match the *granting mechanism* to the permission's risk/reversibility — **automatic grant, trusted-UI (powerbox/picker), runtime prompt, install-time, or explicit confirmation** — explicitly to *cut the number of prompts and reduce habituation*. EFS should adopt this taxonomy as its permission-model backbone.
- **Wijesekera et al., "Android Permissions Remystified" (USENIX Security 2015).** 36-person field study, 27M data points: **80%+ of users wanted to block at least one request; users wanted to block ~⅓ of all requests.** Established *contextual integrity* — prompt at point of use, in context, not at install time. Follow-ups showed even contextual prompts leave users unable to decide well, and many grants are later regretted.
- **Bilogrevic et al. (Google), "Shhh… be quiet!" (USENIX Security 2021) + Chromium quieter-permissions (2020).** Telemetry from **300M+ users / 800M+ prompts / 70k+ sites**: notifications = 74% of all prompts but only ~10% desktop / ~21% Android grant rate — i.e., **mostly unwanted interruptions.** Chrome auto-enrolls abusive sites into a quiet UI (a chip, not a modal) from M84 (2020). Lesson: **most prompts are noise; the default should be quiet/deny, and loud modals reserved for genuinely consequential, user-initiated actions.**
- **"Websites Need Your Permission Too" (CHI 2024).** Users find prompts *most* annoying exactly when they intend to deny; dismissing/ignoring dominates for geolocation + notifications.
- **PEPC / `<permission>` element (Chrome origin trial M126–134, 2024–2025; WICG, cross-browser consensus still open).** A browser-rendered, page-embedded permission button that is click-gated, occlusion-checked, and can't be script-triggered — an attempt to make in-page permission requests *user-initiated and un-spoofable* by having the **browser draw the control**. Mozilla filed a standards position; design still in flux. This is the closest existing analog to what EFS's Shell must do, and worth tracking as the state of the art in "trusted control embedded in an untrusted page."

### 1.8 Warning-design research (what actually changes behavior)

- **Schechter et al., "The Emperor's New Security Indicators" (IEEE S&P 2007).** SiteKey-style **security images fail**: when the site-auth image + HTTPS indicators were removed, **58 of 60 users entered their password anyway.** Positive/personalized trust indicators don't work — users don't notice their absence. This is why EFS must not lean on a "your secret image/phrase confirms this is the real Shell" as the *primary* defense (though it has a narrow role — see §3).
- **Akhawe & Felt, "Alice in Warningland" (USENIX Security 2013).** 25M+ real warning impressions: clickthrough varied hugely by *design*, not just content — Chrome SSL 70% bypass vs Firefox far lower; malware/phishing far "stickier" than SSL. Warnings *can* work.
- **Felt et al., "Improving SSL Warnings: Comprehension and Adherence" (CHI 2015).** The redesign that shipped in Chrome: **opinionated design** (make the safe choice prominent, bury "proceed" under Advanced) lifted adherence **31%→58% (lab) and 37%→62% (field)** — *without* improving comprehension. **Lesson: you win adherence through choice architecture (default to safe, add friction to the dangerous path), not through explaining the threat.**

### 1.9 Passkeys — removing the spoofable secret

- **FIDO2/WebAuthn passkeys (broad 2024–2025 adoption; NCSC, Apple, arXiv census 2026).** The private key never leaves the device; **there is no shared secret to phish**; auth is cryptographically bound to the real origin and only answers server challenges. CISA/NIST classify correctly-implemented FIDO2 as highest-assurance phishing-resistant. 60% of top phishing-target companies have rolled out passkeys; Meta added support Jun-2025. **Lesson: the most robust anti-phishing move is architectural elimination of the secret, not better warnings about protecting it.** EFS already gets much of this free by using wallet/passkey signatures rather than passwords — but it must not reintroduce a phishable secret in the Shell-unlock or prompt-confirm flow.

## 2. WHAT IS EMERGING

- **ERC-7730 clear-signing** moving to Ethereum Foundation governance; v2 (2026) adds cross-chain + software-wallet support + privacy considerations. Becoming *the* structured-decoding standard. EFS should express its envelope/record decoding in a 7730-compatible descriptor form so any wallet can clear-sign EFS writes.
- **PEPC `<permission>` element** — still an origin trial, cross-browser consensus unsettled (Mozilla concerns). The idea (browser-drawn, click-gated, occlusion-checked in-page control) is the exact primitive EFS needs; EFS may have to *build its own* Kernel-drawn equivalent rather than wait for the platform.
- **Transaction simulation as default** (Rabby, MetaMask+Blockaid, Safe post-Bybit tooling) is consolidating into an expected baseline. Tension with EFS's no-ambient-HTTP: simulation needs an endpoint.
- **Locked-mode / keyboard-lock prompts** (Chrome) — browsers tightening fullscreen/keyboard-lock so a site can't trap the user; EFS Shell should never *want* fullscreen-trap patterns and should treat any app request for them as high-risk.
- **Document PiP hardening** — spec continues to forbid position-setting and keep browser chrome, precisely to stop it becoming a spoofing surface.

## 3. EFS-SPECIFIC INVENTION SPACE

Things no existing system gives EFS off-the-shelf, because EFS lives *inside a page* with *no browser chrome to borrow*:

1. **A Kernel-owned render region the Shell draws and Ring-3 apps provably cannot paint into or occlude.** Existing analogs (Qubes borders, iOS picker, PEPC) all rely on a *more privileged layer* (hypervisor, OS, browser) drawing the trusted pixels. EFS's more-privileged layer is the Kernel worker + Bootstrapper, but they render into the *same DOM/GPU surface* as apps. The invention: enforce that Shell chrome is composited in a top-level browsing context / cross-origin isolated frame / dedicated always-on-top surface that app compartments have **no DOM handle to and cannot `requestFullscreen`/PiP over.** This is the crux; see §4 gaps.
2. **A user-configured "Shell secret" as a *negative* tripwire, not a positive trust signal.** SiteKey proved positive personalization fails. But a user-chosen accent color / phrase shown *only in genuine Shell prompts* has narrow value: it doesn't make users trust a real prompt, but its *absence or wrongness* in a fake can be a tell for the minority who look, and it raises attacker cost. Frame it exactly as Chromium/Stark say: negative indicator only.
3. **Structured EFS-action decoding baked into the protocol.** Because EFS writes are EIP-712 envelopes over Merkle roots of typed records, EFS can render **provably-complete, human-readable previews from the records themselves** — a strictly stronger position than generic Ethereum wallets, which reverse-engineer intent from calldata. EFS should ship ERC-7730-style descriptors for every record kind so the preview is deterministic and the *same bytes the user reads are the bytes signed*.
4. **The picker-as-grant powerbox as the primary permission surface**, replacing most prompts (see §4 translation).

## 4. EFS TRANSLATION — opinionated recommendations

**T1. Do not fight the pixel war; move the decision off the spoofable surface.** The Shell's secure prompt must not be "a nicely-styled modal inside the app viewport that says trust me." BitB, modal phishing, and the line-of-death literature guarantee that loses. The secure prompt must render in a **Kernel/Bootstrapper-owned top-level surface** that app SES compartments have no DOM reference to, cannot draw over (no app `requestFullscreen`, no app Document-PiP, no app `pointer`/`keyboard` lock — the Kernel denies these capabilities by default), and cannot occlude. If EFS cannot guarantee such a surface within one browser tab, the fallback is a **separate top-level browsing context** (dedicated origin/window) for security-critical confirmations — accept the UX cost.

**T2. Make the picker the grant. Adopt the powerbox pattern as the #1 permission primitive.** iOS/Android pickers + Felt's taxonomy are the strongest positive evidence in the field: *selection = consent = capability*, no separate prompt, no ambient authority. For EFS: file open/save, "share these bytes," "grant this app this folder," "pick this identity/persona" should be **Shell-rendered pickers that return a scoped, revocable capability handle for exactly the selected object** — no prompt to habituate to. This directly serves the handoff's `openFilePicker` OS-SDK primitive and its capability-by-selection permission model.

**T3. Ruthlessly budget prompts; default to quiet-deny.** The Google 300M-user data and Vista UAC fatigue are unambiguous: modals are mostly noise and over-prompting destroys the security value of the rare important prompt. Reserve full Shell modal chrome for the *consequential, irreversible, user-initiated* checkpoints the handoff already names — **signing/flush, publishing, spending, app install, admin grants, export, local-data deletion.** Everything else = picker (T2), quiet chip, or auto-grant per Felt's taxonomy. A prompt the user didn't initiate should default to the quiet path.

**T4. Never blind-sign. Structured decoding is a protocol-level requirement, not a UI nicety.** Radiant + Bybit + Permit2 = **$2B+ lost to display-vs-signed-bytes divergence.** EFS's advantage: it signs typed records, so the preview can be *derived from and provably equal to* the signed content. Requirements: (a) render every write as human intent from the records themselves; (b) ship ERC-7730 descriptors so third-party/hardware wallets can clear-sign EFS writes too; (c) show **balance/effect diffs** (Rabby model) for anything economic; (d) surface the **Merkle root / envelope hash** and let a hardware wallet or an independent validator recompute it (Safe-Utils lesson). The bytes the human reads MUST be the bytes signed.

**T5. Interaction-gate every consequential confirm; assume the cursor and keyboard are lying.** Copy Chromium's playbook exactly: no default-focused accept button (anti-keyjacking); **activation delay** before the confirm button is live (~1–3s for high-risk, ~500ms otherwise) with `InputEventActivationProtector`-style logic; require the click to land on a freshly-revealed control at a non-predictable location; ignore clicks arriving suspiciously fast after the surface appears (defeats the timed cursorjacking/clickjacking pattern where 16% still fell for it even with cursor customization off).

**T6. Negative indicators over positive; opinionated defaults over explanation.** SiteKey (58/60 ignored the missing image) and the SSL-warning redesign (adherence 31→58% via choice architecture, comprehension flat) settle this. EFS should (a) **not** build a "green checkmark = safe" habit users learn to see and attackers learn to fake; (b) instead warn loudly and specifically when a write is dangerous, an address is unverified/poisoned, a permission is being expanded, a lens is being shadowed, or an app is requesting wildcard network; (c) make the **safe choice the prominent default** and put the dangerous path behind an "Advanced/Proceed anyway" fold.

**T7. Kernel-derive all identity shown in prompts; never render app-asserted trust metadata as trusted.** The modal-phishing/WalletConnect-Verify lesson. A Shell prompt's "app name / icon / origin" must come from the **verified package CID / signer / manifest / capability receipt**, not from strings the app supplied. If identity is app-asserted, label it as unverified in situ.

**T8. Kill address truncation for anything that authorizes value or trust.** Address-poisoning ($83.8M + a single $50M) is a *pure UI-truncation* failure. For recipients/attesters/signers EFS must show full identifiers, checksum/color-hash the full value, warn on look-alike/poisoned entries, and prefer **address-book / picker selection (T2)** over paste-and-eyeball.

**T9. Deny apps the fullscreen/lock/PiP toolkit by default.** Fullscreen + pointer-lock + keyboard-lock + Document-PiP are the exact tools used to erase and repaint trusted chrome. Ring-3 apps get none of these as ambient capability; each is an explicit, high-risk, rarely-granted request, and even when granted the Kernel keeps an always-available, always-advertised escape that returns to trusted Shell (the fullscreen-mitigation lesson).

**T10. Know when to require the real wallet UI / hardware signer.** For the highest-consequence, irreversible actions (large spends, admin/root capability grants, changing wallet policy, key export), the *only* fully defensible move is to route confirmation through an out-of-EFS trusted signer — a hardware wallet doing its own on-device clear-signing (ERC-7730), or the platform passkey/WebAuthn ceremony (which is origin-bound and unspoofable by the page). EFS's Shell prompt then becomes the *preview*, and the *authorization* happens on a surface EFS doesn't draw. Define an explicit risk threshold above which Shell-only confirmation is disallowed.

**T11. Frontend integrity is part of the trusted-path story.** Bybit was a *frontend supply-chain* compromise (no SRI, no integrity alerting). EFS's content-addressed OS profile (Kernel CID + Shell CID) is a strong native answer — the Bootstrapper should verify Shell/Kernel hashes against the signed profile before boot, and refuse/rescue-boot on mismatch. Make "the Shell you're running is the Shell you pinned" a checkable, user-visible fact.

**T12. Lean on passkey/WebAuthn for Shell unlock and re-auth.** No shared secret to phish, origin-bound, page can't spoof the ceremony. Use it for unlocking the Kernel keystore and for step-up on high-risk actions rather than a Shell password/PIN that a fake Shell could harvest.

## 5. WHERE EFS v2 PROTOCOL / ARCHITECTURE CONFLICTS OR UNDER-SUPPORTS THIS

- **G1. There may be no truly unspoofable trusted surface inside one browser tab.** The whole Shell-owned-prompt guarantee assumes a region apps can't imitate/occlude, but SES compartments, the Shell, and app content share one DOM/compositor. Qubes/iOS/PEPC all get their trusted pixels from a *more privileged rendering layer*; EFS's Bootstrapper/Kernel run in workers with **no DOM**. **This needs a concrete rendering-isolation design (top-level context vs cross-origin-isolated iframe vs separate window) before the "apps cannot imitate" claim is credible.** Feed back into the render-isolation / mirror-scheme note.
- **G2. No web secure attention key.** Users can't invoke "show me the real Shell" in a way the page can't fake, unlike Ctrl+Alt+Del. EFS can approximate (a Kernel-drawn indicator, a browser-level PWA install so the OS window frame becomes the trusted boundary, a keyboard shortcut the Bootstrapper registers) but none is as strong as a kernel SAK. **Document this as an accepted residual risk and design the negative-indicator + passkey stack around it.**
- **G3. Transaction-preview simulation vs no-ambient-HTTP.** The strongest anti-blind-signing UX (balance-diff simulation) needs an RPC/simulator endpoint — a network capability and a privacy leak about *what the user is about to sign*. EFS's protocol advantage (previews derivable from the typed records with no network) partly resolves this for EFS-native writes, but *economic side effects on chain* (token transfers, contract state) still need simulation. **Decide: user-configured simulation endpoint as an explicit capability, local light-client simulation, or preview-from-records-only with an honest "on-chain effects not simulated" label.**
- **G4. Permanence + signed-bundle semantics amplify prompt stakes.** The handoff already warns a signed bundle is a portable authored artifact, not a private draft, and "delete = unlist." The secure-prompt surface must make the *irreversibility class* legible at the confirm step, because unlike a normal wallet tx the user cannot rely on "it didn't broadcast yet." **The read-grade/permanence vocabulary needs a matching risk-class vocabulary in the prompt layer** (e.g., "this is permanent," "this is signed and now portable," "this only unlists").
- **G5. Lens/read-grade trust signals are positive indicators — and positive indicators get spoofed and ignored.** LIVE/STALE/REVOKED grades and lens-endorsement are exactly the kind of positive trust UI SiteKey-type research says users tune out. The prompt/chrome design should render these primarily as **warnings on the bad states** (REVOKED, EQUIVOCAL, UNKNOWN-as-not-absence, discovery-not-endorsed) rather than reassurances on the good ones, or they'll habituate. This is a UX-side constraint the read-lens spec should acknowledge.
- **G6. Agent actors and secure prompts.** If agents are first-class actors, the "human-initiated, interaction-gated, activation-delayed" confirm model partly breaks — an agent can't be trusted to satisfy a human-presence gate, and auto-approving agent actions reintroduces the very habituation/blind-signing failure. **Need an explicit rule that the T3/T4/T5 checkpoints (sign/flush/spend/admin/export/delete) are never satisfiable by an agent alone**, only by a human on the trusted surface — which the handoff's Approval Queue anticipates but the prompt-security model must enforce.
