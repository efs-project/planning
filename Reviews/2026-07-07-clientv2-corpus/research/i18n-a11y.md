# Internationalization, input, fonts, and accessibility as OS services — research digest
**Corpus:** 2026-07-07-clientv2-corpus. **Agent lane:** i18n-a11y. **Date:** 2026-07-07.

## Thesis for the OS designers

The browser already ships a large, offline-native i18n stack: **ECMA-402 `Intl.*` reads CLDR/ICU data baked into the JS engine — no network, no fetch, no font/CDN call.** That is a gift for a cypherpunk offline OS: correct locale formatting with zero ambient HTTP is available today, for free, in every engine EFS targets. The catch is that engine-bundled ICU data is *versioned per-browser and per-release*, so formatting is **not deterministic across engines** — which collides directly with EFS's "signed receipts / reproducible citations" needs. The clean architecture is a two-track locale layer: (1) use native `Intl` for ephemeral human display (fast, offline, zero-leak if mediated), and (2) ship a **content-addressed ICU4X-in-WASM formatter + signed CLDR/language/font packs** as the *canonical* renderer for anything that must be reproducible (receipts, timestamps, audit logs, cross-device agreement). The `LocaleHandle` capability is the right instinct: it lets apps say "format for the user" without handing over the fingerprint (language list + timezone + calendar + numbering + installed fonts ≈ 50–60 bits of entropy). Input (IME), fonts, and bidi are where custom UIs quietly break for the non-Latin half of the world; each needs a first-class OS answer, not app-by-app improvisation.

## 1. WHAT EXISTS TODAY — ECMA-402 / `Intl` coverage (offline, engine-bundled)

All of these run against the engine's own bundled ICU/CLDR — **no network** — which is exactly what an offline OS wants. Support reality as of mid-2026:

| API | What it does | Baseline / support | Safari/Firefox lag notes |
|---|---|---|---|
| `Intl.NumberFormat` (v3) | numbers, currency, units, `roundingMode`, `roundingPriority`, grouping, ranges (`formatRange`) | Widely available; v3 features Baseline by ~2023–24 | v3 grew incrementally; older Safari lacks `roundingMode`/`formatRange` — feature-detect |
| `Intl.PluralRules` | cardinal/ordinal plural category selection | Widely available all engines | fine |
| `Intl.RelativeTimeFormat` | "3 days ago" | Widely available all engines | fine |
| `Intl.ListFormat` | "A, B, and C" conjunctions/disjunctions | Widely available all engines | fine |
| `Intl.DisplayNames` | localized language/region/script/currency names | Widely available all engines | fine |
| `Intl.Locale` | parse/maximize BCP-47, expose calendar/numbering/hourCycle | Widely available | `Locale.getCalendars()`-style info methods landed later; feature-detect |
| `Intl.Collator` | locale-aware compare, `usage:"sort"` vs `"search"`, `sensitivity` base/accent/case/variant | Widely available all engines | see §9 — **no substring search** (only whole-string compare) |
| **`Intl.Segmenter`** | grapheme / word / sentence segmentation (CJK/Thai wordbreak, safe truncation) | **Baseline Newly available 2024-04-16** (Chrome 87, Safari 14.1, **Firefox 125**); Widely-available target 2026-10-16 | Firefox was the last major, only in **v125 (Apr 2024)** — recent enough to matter for old installs |
| **`Intl.DurationFormat`** | "1 hr 30 min" localized durations | **Baseline Newly available 2025-03** (Chrome 129, **Firefox 136 / Mar 2025**, Safari 16.4) | Firefox shipped only Mar 2025 — youngest of the set |

**Load-bearing takeaways:**
- `Intl.Segmenter` is the single most important i18n primitive for a *file/graph OS*: `String.length`, `.slice()`, `.split(' ')` all operate on UTF-16 code units and **corrupt emoji/combining sequences** (👨‍👩‍👧‍👦 has `.length === 11`). Any filename ellipsizing, search snippet, comment preview, or path label must truncate on **grapheme boundaries** via `Intl.Segmenter('…',{granularity:'grapheme'})`, and tokenize CJK/Thai/Lao/Khmer (no spaces) with `granularity:'word'`. Naïve `slice` produces lone surrogates like `\uD83D`.
- The whole `Intl` surface is **offline by construction** — it never touches the network, which satisfies the "no ambient HTTP" rule automatically. This is a strong reason to prefer `Intl` over any JS i18n library that fetches CLDR JSON.

## 2. EMERGING — Temporal (dates/times done right, finally shipping)

- **Firefox 139 (May 2025)** was first to ship Temporal on by default; **Chrome 144 (Jan 2026)** shipped it; Edge follows Chromium; **Safari** has most of it in Technical Preview, some under flag. Sources disagree on TC39 stage (Stage 3 "no changes expected" per Bryntum vs. reports of Stage 4 / ES2026) — treat as **shipping-but-not-yet-universal**; Safari stable is the gating dependency.
- Relevance: Temporal replaces the broken `Date` for timezone-correct, calendar-aware timestamps — important for an OS that renders **chain timestamps** and **audit logs** across timezones. But the polyfill is heavy and the **IANA tz database** is a moving target; if EFS renders a "canonical time", it must pin the tz data version (see §14/§15).

## 3. Messaging — MessageFormat 2.0 (MF2) and `Intl.MessageFormat`

- **MF2 is spec-stable as of Unicode CLDR 47 / LDML Part 9, released 2025-03-13.** Unicode's blog states the *entire* spec — data model, syntax, processing, conformance — reached **Stable** with stability guarantees; implementations can finalize APIs. This is the modern replacement for ICU MessageFormat and for gender/case/plural-rich natural translations.
- **`Intl.MessageFormat` (TC39/ECMA-402 proposal) is NOT shipping** — stuck around **Stage 2 / 2.7**, gated on evidence of ~12 orgs using MF2 in production. So native browser MF2 is **not available**; you use a JS library (`messageformat` 4.x provides an MF2 formatter + a polyfill of the proposed runtime API).
- i18next added MF2 support in 2025–26 but migration is early. **Do not bet the OS's string layer on native `Intl.MessageFormat`.** Adopt MF2 *syntax* (future-proof, Unicode-stable) via a small bundled JS runtime, and keep the door open to swap in native later.

## 4. The deterministic path — ICU4X in WASM

- **ICU4X 2.0 released 2025-05-29** (Unicode). Rust, `no_std`-friendly, **50–90% smaller binary/memory footprint than ICU4C**, purpose-built for constrained/embedded/web. Ships **JS/TypeScript bindings on npm** (via WASM/FFI) and C/C++.
- **Data slicing is the headline feature.** Three providers: **Baked** (compile-time), **Blob** (`icu_provider_blob`, runtime-loadable byte blob), **Filesystem**. `icu4x-datagen` slices by component *and* locale. Concrete numbers from the 2.0 tutorial: a datetime blob is **~5 MB with `--markers all`**, drops to **~19 KB** after `--markers-for-bin` binary analysis, and **~8 KB** for a single calendar-specific formatter (of which ~5 KB is shared locale-fallback data). A "kitchen-sink all-locales-all-markers" WASM bundle is ~8 MiB; a **sliced, per-need** bundle is kilobytes.
- Why this matters for EFS: ICU4X gives you a **version-pinned, content-addressable, deterministic** formatter you control — the same input + same data blob = same output on every device, unlike engine `Intl`. This is the correct engine for **canonical receipt/timestamp rendering** and for locales the host engine doesn't bundle. Components available: datetime, collation, segmentation, plural rules, number formatting, list/display names.

## 5. CLDR data packaging strategy

- CLDR is the source of truth (releases ~2×/yr; **CLDR 47 = 2025-03**). Two packaging models to combine:
  1. **Lean on engine ICU** for common locales displayed ephemerally (free, offline, but non-deterministic across engines/versions).
  2. **Ship sliced ICU4X blobs** as **signed, content-addressed "language packs"** for (a) canonical rendering and (b) locales the engine lacks. Slice per-locale + per-component so a user who only reads `ar`+`en` never downloads `zh` collation tables.
- Package granularity should mirror the font strategy (§6): one pack per locale/script block, content-addressed, cached in OPFS, verified by hash before use, rolled back like any OS generation.

## 6. Fonts for an offline OS

**The size math is dominated by CJK.** Facts: Noto covers **166 of 172 Unicode 17.0 scripts (Nov 2025)** across **~180 families / ~2,300 fonts**; the full collection is ~1 GB+ uncompressed. Per the W3C i18n note, a single **Han font is ~8 MB** and impractical as a web font — Noto Sans CJK per-language weights are tens of MB; a pan-CJK variable font is larger still. WOFF2/Brotli helps (a subsetted Latin WOFF2 can be ~12 KB vs 168 KB TTF, a ~93% cut), but **subsetting does not shrink CJK meaningfully** because CJK users need most of the glyphs.

Strategy for EFS:
- **Tiered, content-addressed font packs.** Ship a small **core** (Latin+Cyrillic+Greek+Arabic+Hebrew+Devanagari+Thai + a UI weight) — a few MB — as the default offline set. Treat **CJK, Korean, and full emoji** as **on-demand signed packs** the user (or locale) pulls once and caches. One pack per script block, not one monolith, for cache efficiency.
- **Variable fonts** consolidate weights/widths into one file (shared outlines interpolated) — good when the UI uses multiple weights; **not always smaller** than a single subset static weight. Use one variable UI face + static fallbacks.
- **`unicode-range` controls *which* file downloads, not file size** — the whole matched file loads if any char is in range. Pair physical subsetting with `unicode-range` so a page of pure Latin never pulls the CJK pack.
- **`local()` and system fonts are a privacy landmine.** Safari/WebKit **restricts `local()` and non-system font-family names** (since Safari 12, 2018) specifically to kill font fingerprinting — but that broke rendering for minority-script users (Javanese content became "completely unreadable" per the W3C note). Lesson for EFS: **do not depend on user-installed fonts** for correctness; ship your own packs. If you ever expose local fonts, gate behind a **font privacy budget** (penalize probing many fonts) and per-user opt-in, per CSS Fonts 4 / W3C guidance.
- **No hidden HTTP font loads, ever** (Google Fonts CDN etc. is an observer). All fonts are OS packs. This is already a stated EFS non-negotiable; the font layer must enforce it.

## 7. Text rendering — RTL / bidi / mixed-direction identifiers

- Bidi is a **security surface**, not just layout. The Unicode Bidi Algorithm + override controls (U+202E RLO, and the isolates RLI/LRI/FSI/PDI added in Unicode 6.3) can **reorder visible text to spoof URLs, filenames, and identifiers** ("BiDi Swap" — still exploitable a decade on; browsers have no comprehensive fix as of 2025–26). "Trojan Source" attacks used the same trick to hide code semantics.
- **EFS-critical case: mixed-direction identifiers.** An address like `0xABC…def` next to Arabic/Hebrew label text, or a vanity path mixing scripts, will visually reorder unpredictably and can be **weaponized to make one address look like another** or hide a suffix. Mitigations the OS must bake in:
  - Wrap every user-controlled string in **bidi isolation** (`<bdi>` / `unicode-bidi: isolate` / RLI…PDI) so it cannot leak direction into surrounding chrome. Never concatenate an untrusted label into a sentence without isolation.
  - Render **security-critical identifiers** (addresses, hashes, path segments) in an **explicitly LTR, isolated, monospace** context, ideally chunked (`0xABCD 1234 …`), and consider stripping/again-flagging bidi-control characters and mixed-script confusables in identifier positions.
  - Apply **Unicode confusable/mixed-script detection** (UTS 39) to path names and display names, and surface a warning when an identifier mixes scripts.

## 8. Input & IME — the custom-editor problem

- **`contenteditable` and custom editors break IME.** Composing Chinese/Japanese/Korean/Vietnamese requires the OS IME composition window to attach to a real editable region; canvas/div editors historically mishandle `compositionstart/update/end`, dead keys, and candidate windows.
- **`EditContext` API** is the modern fix — decouples text input/IME/composition from DOM rendering so you can build a rich editor (even canvas) that still gets native IME, emoji picker, and OS input surfaces. **BUT it is Chromium-only: shipped Chrome/Edge 121 (Jan 2024); Firefox and Safari have NOT shipped** (Mozilla: positive-ish but a11y/compat concerns; WebKit: no signal). So EditContext is **not a portable foundation** in 2026.
- **Practical stance for EFS:** prefer **native `<input>`/`<textarea>`/`contenteditable`** (which get IME for free) for text entry; only reach for EditContext-class custom editors where unavoidable, and **feature-detect + fall back**. Do NOT ship a canvas/WebGL text editor as the default note/comment surface — you will silently exclude CJK/IME users on Firefox/Safari.
- **Virtual keyboard on mobile is broken at the API level.** `navigator.virtualKeyboard` (overlaysContent, `geometrychange`, `env(keyboard-inset-*)`) is **Chromium-Android-only** and even there has real bugs (height overshoot during animation, wrong coordinate space off by ~368px, misimplemented CSS insets). **Safari and Firefox have never shipped it** (6+ years; Mozilla says not an interop priority for 2026). The **portable** answer is the **`interactive-widget` viewport meta key** (`resizes-content` / `resizes-visual` / `overlays-content`; Chrome 108+, **Firefox 132+**) plus **`window.visualViewport`** resize/geometry for layout. Design mobile Shells around visualViewport, not VirtualKeyboard API.

## 9. Collation & search offline

- `Intl.Collator` gives locale-correct **sorting** and whole-string **matching** (`usage:"search"`, `sensitivity` levels) offline — good for ordering file lists and accent/case-insensitive equality.
- **Gap: no locale-aware substring search.** ECMA-402 issue #506 (`startsWith/endsWith/includes/indexOf` on Collator) is still open/unshipped. For "find text in file/graph" you cannot get diacritic/case-insensitive *substring* matching from the platform. Options: **ICU4X collation + segmentation in WASM** for a real locale-aware search index, or a pragmatic **NFKD + base-folding** normalization pass for a "good enough" fold-insensitive search. Locale-aware tokenization for indexing (CJK/Thai) again leans on `Intl.Segmenter`/ICU4X — offline-capable.

## 10. Accessibility standards status (2026)

- **WCAG 2.2 is the stable target** — published 2023-10-05, updated 2024-12-12, now **ISO/IEC 40500:2025**. It adds **9 success criteria**; the ones that bite an OS:
  - **2.5.8 Target Size (Min) — 24×24 CSS px** (now AA, was AAA). Dock icons, window controls, chips, permission buttons must meet it.
  - **2.4.11 Focus Not Obscured** — sticky Shell chrome / toolbars must not hide the focused element.
  - **2.5.7 Dragging Movements** — any drag (window move, file drag, reorder lens list) needs a single-pointer non-drag alternative.
  - **3.3.8 Accessible Authentication** — no cognitive-function test in auth; **must allow paste + password managers**. Directly relevant to **seed-phrase / wallet / passkey** flows: do not block paste of a mnemonic or a signed bundle.
  - **3.2.6 Consistent Help**, **3.3.7 Redundant Entry** — help placement stable; don't re-ask for info already given.
- **WCAG 3.0 is an incomplete Working Draft** — newest draft **2026-03-03 with 174 requirements ("outcomes")**, replacing A/AA/AAA with graded scoring across 12 functional categories. It will **coexist with, not supersede, 2.2**; final Recommendation is **~2028–2030**. **Build to 2.2 now; watch 3.0.** Do not chase 3.0 conformance yet.

## 11. ARIA + Web Components reality (Lit / Web Awesome / Shadow DOM)

This is the sharpest a11y trap for EFS's chosen stack (Lit + Web Awesome, heavy Shadow DOM).

- **Shadow DOM breaks ARIA cross-root references.** `aria-labelledby`, `aria-describedby`, `aria-activedescendant`, `aria-controls` **cannot cross a shadow boundary by IDREF** — a `<button aria-labelledby>` in the light DOM cannot point at a `<label>` inside a component's shadow root, and vice versa. This silently breaks accessible names/relationships in component-heavy UIs (Nolan Lawson / Igalia have documented this for years).
- **What works today:** **`ElementInternals`** (all major engines since **Firefox 119, Oct 2023** for ARIA-mixin string reflection) lets a custom element set **default ARIA role/state** (`this._internals.role='button'`, `ariaLabel`, etc.) on its host — the correct way to give web components implicit semantics. ARIA *element references* may cross shadow boundaries **only via ElementInternals**, which covers host default semantics but **not** general cross-root wiring.
- **Emerging fix: "Reference Target for Cross-Root ARIA"** (`shadowrootreferencetarget` / ElementInternals reference targets), superseding the older delegation+reflection proposals. **Prototyping in Chromium and WebKit; Chromium origin trial as of May 2025.** Not Baseline; not in Firefox stable. So in 2026 you **cannot rely** on portable cross-root ARIA.
- **Testing blind spot:** automated a11y tools **cannot see ARIA semantics applied via `ElementInternals`**, producing false positives and undetected gaps. The **2026 WebAIM Million** found pages using ARIA averaged **59 detectable errors vs 42 without** — ARIA is routinely misapplied. Manual screen-reader testing is mandatory.
- **Implication for EFS:** either (a) keep interactive labels/relationships **within a single shadow root** per component, (b) reflect names via `ElementInternals`, or (c) use light-DOM slotting where cross-root wiring is needed — until Reference Target ships everywhere. Web Awesome claims built-in ARIA/labeling + localization, but **do not trust the library blindly**; audit each component with a screen reader.

## 12. Canvas / custom widgets and screen readers

- **Canvas content is invisible to screen readers** — no DOM, no semantics. If EFS renders any content viewer, chart, or editor on canvas/WebGL, it needs **DOM fallback content** inside `<canvas>` and/or a parallel ARIA-annotated DOM mirror. **Prefer SVG over canvas** for interactive custom widgets — SVG has native a11y semantics and takes ARIA + JS interactivity directly.
- ARIA communicates **semantics, not behavior**: a `role="button"` div is not keyboard-operable until you add key handlers + focus management. Every custom Shell widget needs explicit keyboard support, roving tabindex, and focus-visible.

## 13. `prefers-*` media queries (foundation, ship day one)

Well-supported and cheap; wire them into the Shell theme system from the start:
- **`prefers-reduced-motion`** — widely supported; gate all animation/transition.
- **`prefers-color-scheme`** — universal; light/dark.
- **`forced-colors`** (Windows High Contrast) — ~93% support (H2-2024); test Shell chrome, don't override system colors.
- **`prefers-contrast`** — growing support; offer high-contrast tokens.
- **`prefers-reduced-transparency`** — newest, Chrome-led; honor for glass/blur effects.
- **`prefers-reduced-data`** (if present) — ties into the offline/pack-download decisions.
Note: these queries are themselves **passive fingerprinting signals** (part of the ~50–60 bits of CSS entropy). They're low-risk to honor (read-only, user-set), but the OS should treat the *combination* as sensitive (see §14).

## 14. Offline / local translation

- **Firefox Translations (Bergamot / Marian NMT compiled to WASM)** runs **fully on-device, offline, no cloud** — models downloaded per language pair and cached. This proves **local NMT in WASM is viable in a browser**. But it is **Firefox browser-chrome only**; it is **not a web-content API** EFS apps can call. EFS would have to bundle a Bergamot-class WASM engine + signed model packs itself to offer offline translation as an OS service.
- **Chrome built-in Translator + Language Detector APIs** (web-content APIs, `Translator`/`LanguageDetector`): **stable in Chrome 138 (2025)**, 40+ languages, model downloaded on first use and then **works offline & client-side**, download status hidden for privacy. **Desktop-only, Chromium-only, NOT Baseline** (absent in Safari/Firefox, no mobile). Useful as a **progressive enhancement**, never a dependency.
- **EFS stance:** treat translation as an **optional, content-addressed WASM service + signed model packs** (Bergamot-style) for a real offline guarantee; opportunistically use Chrome's built-in API where present. Never route text to a cloud translator without an explicit network capability + data-sharing warning (translation reveals exactly what the user is reading).

## 15. LESSONS & TRAPS from deployed systems

1. **Engine `Intl` is offline but non-deterministic.** CLDR data differs across Chrome/Firefox/Safari and across releases; the *same* date/number can format differently. Never hash or sign localized output; never assume two devices render a timestamp identically. (This is the core collision with EFS receipts.)
2. **Grapheme naïveté corrupts data display.** `String.length`/`slice` break emoji/combining marks → mojibake and lone surrogates. Every truncation must use `Intl.Segmenter`.
3. **Bidi is an attack surface.** RLO/override + isolates enable URL/filename/identifier spoofing (BiDi Swap, Trojan Source) — unpatched in browsers for a decade. Mixed LTR-address + RTL-label is a real phishing vector. Isolate everything; special-case identifiers.
4. **`local()`/system-font access = fingerprinting.** WebKit disabled it and broke minority-script rendering — the cautionary tale that privacy fixes can destroy i18n if you depend on user fonts. Ship your own fonts.
5. **Locale is a fingerprint.** `navigator.languages` + `Intl.DateTimeFormat().resolvedOptions().timeZone` + calendar + numbering + installed fonts ≈ 50–60 bits and, worse, **inconsistencies** (ru-RU UI on uk-UA system, or a timezone/IP mismatch) are *themselves* a signal. A "format for user" API must not hand apps the raw profile.
6. **Custom editors exclude IME users.** contenteditable/canvas editors mishandle CJK/Vietnamese composition; `EditContext` is the fix but Chromium-only. A non-Latin user hitting a broken editor is fully blocked, not degraded.
7. **VirtualKeyboard API is a mirage.** Chromium-Android-only and buggy; Safari/Firefox never shipped it. Mobile layout must use `visualViewport` + `interactive-widget`.
8. **Shadow DOM silently breaks ARIA relationships.** Cross-root `aria-labelledby`/`activedescendant` don't work; component libraries look accessible but aren't. Reference Target isn't universal yet. Automated tools miss ElementInternals semantics.
9. **ARIA misuse makes things worse** (59 vs 42 errors, WebAIM 2026). Prefer native semantics; add ARIA surgically; manual SR test.
10. **Fluent's lesson:** asymmetric localization (per-locale plural/gender/case complexity in isolation) is powerful and is the right *conceptual* model, **but** the `.ftl` syntax is hard for translators and breaks standard TMS tooling → runtime errors. If EFS adopts MF2 (the standardized successor idea), invest in tooling/validation or translators will ship broken messages.
11. **Canvas = a11y void.** Anything on canvas needs a DOM shadow for screen readers; prefer SVG.
12. **CJK/emoji font weight dwarfs everything.** An "all languages offline" bundle is impractical as one blob; tier and lazy-load or you ship a ~1 GB OS.

## 16. EFS TRANSLATION — concrete recommendations for client v2

**A. `LocaleHandle` as a mediated capability (adopt).**
Apps get a Kernel/Shell-owned `LocaleHandle` exposing *methods*, not *data*: `format.date(x)`, `format.number(x)`, `format.list(xs)`, `collator()`, `segment(str,granularity)`, `plural(n)`, `direction(str)`, `t(mf2Id, args)`. It returns *formatted strings* and *comparison results* without disclosing the user's language list, region, timezone, calendar, or font set. Full-profile access (`getLocaleProfile()`) is a **separate, prompted, high-sensitivity capability** with a fingerprint warning. Default reveal to Ring-3 apps: **only the primary display language tag** (e.g. `en`), coarsened, if anything.

**B. Two-track rendering: display vs canonical.**
- *Display track*: native engine `Intl` (fast, offline, zero-fetch) for ephemeral UI.
- *Canonical track*: a bundled, version-pinned **ICU4X-WASM formatter + hash-pinned CLDR pack** used whenever output must be reproducible — **signed receipts, chain timestamps, audit-log entries, citation/proof links**. A canonical render carries `(cldrVersion, tzVersion, formatterCID, localeUsed)` so any device reproduces it byte-for-byte. **Rule: canonical artifacts always store the raw machine value (ISO-8601 instant, integer, address) and render locale on top; never store only the localized string.**

**C. Signed content-addressed language packs & font packs (adopt the handoff instinct).**
- **Language pack** = signed, content-addressed bundle of: MF2 message catalogs, sliced ICU4X data blobs (per-locale, per-component), help/docs. Cached in OPFS, hash-verified before load, rolled back like any OS generation, **no HTTP fallback**.
- **Font pack** = tiered, content-addressed: a few-MB **core** (Latin/Cyrillic/Greek/Arabic/Hebrew/Devanagari/Thai + emoji subset) shipped by default; **CJK/KR/full-emoji** as on-demand signed packs keyed by the user's locale/content. `unicode-range` + physical subsets so unused scripts never load. Missing-glyph fallback must be honest (tofu box, not silent blank).

**D. Bidi & confusable safety for identifiers (invent + enforce).**
A Shell primitive `<efs-identifier>` (or a mandatory formatting rule) that renders addresses/hashes/path-segments **LTR-isolated, chunked, monospace**, strips/flags bidi controls, and runs **UTS-39 mixed-script/confusable detection**, surfacing a warning badge when a name mixes scripts or embeds direction overrides. Every user-controlled label elsewhere is `unicode-bidi: isolate` by default in the design system.

**E. Input: native-first, IME-safe.**
Default text entry uses native form controls / contenteditable (free IME). Reserve EditContext-class custom editors for genuine rich-editing needs, **feature-detected with a native fallback**. Mobile Shell layout is driven by `visualViewport` + `interactive-widget=resizes-content`, **not** the VirtualKeyboard API. Never ship a canvas text field as a default input.

**F. Accessibility as a foundation service (adopt).**
- Build to **WCAG 2.2 AA**: 24×24 targets, focus-not-obscured under Shell chrome, drag alternatives (lens reorder, window move, file drag), **paste allowed in seed/bundle/wallet fields** (3.3.8).
- Wire all `prefers-*` queries into design tokens day one (motion, color-scheme, forced-colors, contrast, reduced-transparency).
- Web-component a11y discipline: default semantics via **`ElementInternals`**; keep ARIA relationships **within one shadow root** or use light-DOM slotting; plan to adopt **Reference Target** when Baseline. **Mandate manual screen-reader testing** — automated tools can't see ElementInternals semantics and over-flag ARIA. Prefer **SVG over canvas** for custom widgets; DOM fallback for any canvas.
- Screen-reader language tagging: set `lang`/`dir` on every content region from EFS content-language metadata (see gap G3).

**G. Offline translation as an optional, sovereign service.**
Bundle a Bergamot-class NMT-WASM engine + signed model packs for a *real* offline translation guarantee; opportunistically use Chrome's built-in `Translator` where present. Cloud translation only behind an explicit endpoint capability + "this reveals what you're reading" warning.

**H. Locale fingerprint budget (invent).**
Treat the locale profile like the network layer: **default-deny disclosure**. Track a per-app "locale entropy budget"; each additional revealed field (timezone, calendar, font probe, full language list) spends budget and may prompt. The Locale/Network/Privacy centers in the handoff should show a **locale privacy receipt** analogous to the network privacy receipt.

## 17. Where EFS v2 protocol design under-supports the client OS (feedback into `Designs/efsv2/`)

**G1 — Records have no first-class content-language / direction metadata.** The v2 kind-set (TAGDEF/DATA/LIST/PIN/TAG) and `codex-kinds` reserved-key table do not appear to define a **canonical place for BCP-47 language tag + base direction** on DATA files, LIST collections, comments, TAGDEF namespaces, or app manifests. Without it, the OS cannot set correct `lang`/`dir` for screen readers, cannot pick the right font pack/segmenter, and cannot safely render mixed-direction content. *Recommendation:* reserve a `lang` (BCP-47) and optional `dir` (`ltr`/`rtl`/`auto`) key in the reserved-key table (or a conventional TAG schema), so language/direction travel with the signed record and are reproducible.

**G2 — Signed/permanent artifacts vs. non-deterministic localized rendering.** Read grades and receipts assume a stable, reproducible truth, but engine `Intl` output is **not reproducible across devices** and CLDR/tz data changes over time. A "receipt" or "citation" that shows a localized timestamp/number is **not byte-reproducible** and can differ between the author's view and a verifier's view. *Recommendation:* protocol/SDK convention that **canonical values are stored raw** (ISO-8601 instant + explicit tz, integer/decimal, raw address) and that any localized form is a *client-side lens* carrying its `(cldrVersion, tzVersion, formatterCID)` — never signed as truth. The read-lens-spec's freshness/venue vocabulary should gain a parallel note that **rendering locale is a lens too**, honestly labeled.

**G3 — `expiresAt` / STALE vs. locale-data staleness.** Read grades track record freshness, but there is a second, invisible staleness axis: the **language/font/CLDR pack version** used to render. A user on an old signed OS generation may render an address label or plural form differently than a fresh one, or lack a script's font entirely (tofu). *Recommendation:* the OS profile/generation (Nix-flake-style closure) should pin locale-pack + font-pack + tz-data CIDs alongside Kernel/Shell CIDs, and the read-grade UI should be able to say "rendered with locale pack N" when it affects safety-critical display (identifiers).

**G4 — Confusable/mixed-script identifiers at the protocol edge.** Vanity paths and TAGDEF namespace keys are user-chosen strings that can embed bidi controls or confusable/mixed-script characters, enabling **path/name spoofing** that the "self-certifying address is the root" doctrine doesn't fully neutralize (a human still reads the vanity label). *Recommendation:* SDK-level canonicalization/validation guidance (UTS-39 restriction levels, bidi-control rejection or flagging) for path segments and namespace keys, and a client rule to always show the raw self-certifying root next to any vanity/mixed-script label.

**G5 — MF2 catalogs as content-addressed records.** If translation packs and app-manifest strings are to be signed and offline (handoff requirement), the protocol needs a blessed pattern for **language packs as DATA/LIST records** (MF2 catalog + locale metadata), with lens-curated trust so a user can choose whose translation of an app/namespace they trust. *Recommendation:* an `apps-cookbook`-style blessed pattern for "signed language pack" and "signed font pack" records, including how a lens endorses a translation set.
