## The mission ends (the only fence)

EFS is an on-chain **filesystem *and* graph database**: a permanent (100-year) archive, credibly neutral, verify-don't-trust, cypherpunk, hyperlinkable, that people build real apps on. It runs on a **native kernel** (not EAS): records are chain-free EIP-712 Merkle-signed envelopes; the signature is the authorization; the recovered signer is the author; anyone relays or replicates. Reads are **per-viewer lenses** (trust is the reader's, first-attester-wins). Portability is **replication** (copy the signed records + bytes to another chain and read them natively) — not cross-chain proofs, and deliberately *no* portable cross-chain "currency" (no global is-this-latest / is-this-revoked). These ends are the fence. A filesystem feature that needs mutation, deletion-as-destruction, write-permissions, exclusion, or a trusted global clock is pushing against a grain — and that friction is usually where the design work actually is.

## The settled base a filesystem pass builds on

This is the substrate — accurate as of today, each item traceable to a ruling doc (see the appendix pointers). Treat it as the floor you stand on, not a ceiling.

**Data model — five kinds, two ops.** [settled]
- **TAGDEF** — the unowned namespace: paths, folders, categories, property keys. Permanent, unrevocable. A shared Schelling point: `tagId = keccak(DOMAIN, parentTagId, keccak(name), kindTag)`, so anyone who spells the same path derives the same node. The *kind word is in the derivation* (one parent can hold the same name under different kinds as distinct ids). Carries NFC canonical-name validation + path permanence.
- **DATA** — owned file identity (author+salt in the preimage → unforgeable, unsquattable). Permanent. **A file's identity is its DATA record, not its bytes** (ADR-0049) — on-chain and off-chain files are equally real.
- **LIST** — owned collection charter (`appendOnly`, `targetKind`, `maxEntries`); config folds into `listId` so same-id-different-charter can't exist.
- **PIN** (cardinality-1) and **TAG** (cardinality-N, weighted) — the edge kinds. Both revocable claims, in **REF** layout (→ an object id) or **VAL** layout (→ a value, ≤8192 bytes, string-only, auto-interned by the kernel). Cardinality is part of slot identity — a frozen invariant; merging PIN/TAG is refused.
- Ops: **ASSERT(0) / REVOKE(1)**. (PROPERTY / MIRROR / REDIRECT / LIST_ENTRY were deleted and re-homed onto the above — they are *not* part of the model.)

**The mutation primitive — slots.** [settled] A **slot** is a `(author, key)` position. There is no shared mutable cell: not "the `/plan.md` slot" but *Alice's* slot and *Bob's* slot. Per-slot winner = `argmax by (order, recordDigest)` (last-write-wins). **Empty-on-revoke**: if the winner is revoked, the slot reads EMPTY (no fallback to a prior claim, no resurrection; the author re-asserts to refill). Superseded claims stay reachable (disposition SUPERSEDED, never silently absent); `getSlot` returns `supersessionCount + priorClaimId` in O(1). Slot state is a pure function of the admitted set — identical on any venue holding the same records in any order.

**Writes are permissionless.** [settled] Anyone can ASSERT any edge targeting anything; nobody needs permission, because writing never mutates anyone else's view. The byte pool is permissionless by ruling; filtering is edge-only (lenses / gateways / operators). **There is no write-permission model at all** — control lives entirely at read time.

**Reads — lenses + a closed grade vocabulary.** [settled] A lens is an ordered trusted-author list (first-attester-wins) plus deny-sets that *subtract after* resolution (advisory, client-side — "everyone EXCEPT" is a read-fact, never a write-gate). Grades are normative and closed for v2: position states PRESENT / **PROVEN-ABSENT** (the only state that yields fallthrough) / UNKNOWN (STOP — never fall through); claim dispositions EQUIVOCAL → CONTESTED → REVOKED → STALE → SUPERSEDED → LIVE; currency HOME-LIVE / AS-OF(N) / UNKNOWN-CURRENCY. **Read-lens-spec is Durable (iterates freely), not Etched** — its algorithms and grade-rendering are cheap to change; only the *pins* it names (the envelope/kind/derivation surfaces) are frozen.

**Currency — the one lever.** [settled] `expiresAt` is the last word of every claim body; the kernel stores it opaquely and **never checks a clock at admission** (clock-free admission is a master invariant). Semantics: **stale-not-dead** — expired-not-revoked keeps authenticity, voids currency; STALE is always distinct from REVOKED. No portable cross-chain currency exists; safety-critical apps use author-set expiry.

**Move / symlink / mirrors — what already exists.** [settled] Rename/move = `movedTo` PIN (auto-follow, renders a moved-from breadcrumb). `symlink` PIN (auto-follow). `sameAs` / `relatedVersion` TAGs (never auto-followed). `supersededBy` (dual-role). Combined follow budget `MAX_AUTO_FOLLOWS = 8`, cycle-detected. ~13 reserved-key rows carry the re-homed enforcement of the deleted kinds. Folder visibility derives from a read-time parent-walk.

**Enumeration — the discovery index.** [settled mechanism, pending James's gas sign-off] Container-scoped per-tagId, bounded, paginated (`discover(tagId, cursor, limit≤256)`), admission-ordered, DISCOVERY-flagged. **Enumeration ≠ endorsement**: every entry must pass lens-grading; counts are never GATE-consumable; discovery output never satisfies a PROVEN-ABSENT check. Backed by an `allClaims` append-only spine (~22–27k gas/record) with full bodies in state.

**The structural fact that bends every classic FS feature.** [settled, and worth internalizing before anything else] Classic filesystems have *one mutable object per name*; almost every FS feature (locking, ACLs, quotas, snapshots, undo) is machinery for coordinating many writers around one shared mutable cell. **Tag-core has no shared mutable cell.** So the classic frame inverts: "who may write the object?" → "everyone writes; whose writes does a reader's lens honor?"; "how do writers not clobber?" → "they can't — separate author slots — how does a reader reconcile N authors?"; "delete the file" → "you can't destroy the permanent object; you revoke your edge, and the bytes may persist forever elsewhere." The sharp meta-question that runs through the whole feature space: **which classic FS features are essential semantics users will demand (and must be re-homed onto lens + edges + curation), versus which are artifacts of the one-mutable-cell architecture that simply evaporate here and should be declared *gone* rather than simulated?** A lot of the design is that sorting. The systems that already live natively in this inverted world — **git** (immutable objects, one mutable ref, explicit human merge), **Datomic** (immutable facts + time, "the database is a value"), **RDF/triplestores** (per-speaker assertions, truth as a provenance question), **Perkeep** (permanodes + signed claims, nearly isomorphic to DATA + PIN/TAG) — are worth holding in mind throughout. EFS is closer to *Datomic-facts-under-RDF-provenance-with-git-refs* than to POSIX.

## The time model, as current thinking to pressure

James and I refined the ordering/time story on 2026-07-08. It has been adversarially verified once (verdict: **sound-with-fixes** — the fixes are spec/integration obligations, not model breaks), but the *shape* is still a lean, and a from-scratch designer might land somewhere cleaner. It is squarely in play. Three separated concepts:

- **`order`** (renamed from `seq` — the rename is part of the current thinking) — one value **per signed envelope** (per signature / per batch, *not* per record). A portable ordering key: supersession is LWW by `(order, recordDigest)`; higher `order` wins a slot; revoked reads EMPTY. Deliberately **not** a timestamp (its meaning is purely ordinal, though it's *generated* from the device clock as a 64-bit TID with 10 device bits and capped at ~now+600s to kill "max-it-out-to-win-forever"), **not** a nonce (a nonce forbids replay; EFS *wants* replay — cross-chain copy is a legal, identity-preserving re-submit), **not** an identity input (time must stay out of ID derivation — the original EAS sin).
- **`claimedAt`** (new, optional, **per record / per action**) — the user's *untrusted* claim of when they did that action. It exists because batching collapses many actions under one `order`; `claimedAt` recovers per-action time for timelines/journals. Never proof, never read by the kernel. Its status is genuinely **[open]**: blessed reserved field vs. app-layer convention (same row-vs-convention-vs-reject question the P2 reserved-key candidates face — decide it *with* them).
- **`admittedAt`** (chain-stamped, **per claim**, trustworthy-but-local) — `block.timestamp` at admission, stored `getProof`-provable (the client-OS P1 ask). The *only* trustworthy clock — but **per-chain** (a replica gets a different one), which is exactly why supersession orders on `order`, not chain-time.

The honest one-liner: *the author-asserted timestamp is untrusted; the trustworthy clock is admission — per-chain, venue-labeled, never global; EFS reads need only per-author order.* Two lenses from the prior-art survey worth carrying: this is an **HLC-family** construction (steal HLC's bounded-drift rigor for the hand-picked 600s / 10-bit numbers), and it maps cleanly onto **bitemporality** (`claimedAt` = valid-time, `admittedAt` = transaction-time, `order` = a portable revision key that exists *only because* transaction-time can't travel across chains). That last framing raises a real question: is the clean model two canonical temporal axes plus one portability patch — and if cross-chain transaction-time could travel, would `order` evaporate? Chase or refute it. **Freeze-sensitivity you must respect even while overturning:** `order` lives inside the signed envelope struct and `claimedAt` would live inside the signed record body — both on the format surface bound by the one-final-freeze pledge. So whatever you conclude, the *reserved shape* (is there a per-record time word? how wide? optional?) must converge before the ceremony even as semantics keep iterating.

Why this sits in a *filesystem* kickoff: a "sort by time / show modified date / timeline / restore-as-of" feature lands squarely on all three concepts, and the field is being renamed precisely *because* apps kept mis-trusting it as a clock. The blessed multi-writer collaboration pattern (below) is downstream of the `admittedAt` decision — "whose edit is newer" across authors is only trustworthy as admission order, never author-claimed time. Flag that dependency early; it's freeze-sensitive.

## The filesystem / graph-DB feature space — terrain, not tasks

What follows is a **map of the territory**, not a route through it. Each cluster is a place worth standing and looking around. Where I state an answer it's a foothold to push off from, not a ruling to defer to; the bias throughout is surfacing good questions over supplying answers. A fuller cartography with prior-art annotations for each cluster lives in the corpus ([[fs-feature-space]]) — this is the orientation.

- **Write-sharing / access control / delegation.** Read-side access control is native and strong (a lens is an author allow-list; delegation-of-*trust* composes by nesting lenses). *Write*-sharing in the POSIX sense has **no referent** — Bob can already write; nobody can stop him. "Grant Bob write access" decomposes into at least four different wants (show Bob in the canonical view = curation; write *as* the team = delegated authorship, the hard one, no `act`/on-behalf credential today; exclude everyone else = a read-fact not a write-gate; revoke Bob later = delegated revocation, doesn't exist, `revoker == author`). The sharp question: is "write permission" the wrong primitive to port at all — should the cluster be reframed as "curated-view membership + delegated authorship + deny-subtraction," loudly, so devs stop reaching for `chmod`? Prior art: **Tahoe-LAFS** (caps not ACLs), **UCAN / macaroons** (attenuable delegable offline credentials — *the* decentralized-web answer to exactly this gap), **Solid** (server-side WAC vs. EFS read-side lenses — the exact fork).

- **Multi-writer collaboration** — where the single-author model strains most, and the scope doc flags it hardest ("CRDTs were dismissed early; the shared-folder story needs a real check"). Two shapes are native: **accumulation** (many additive authors — threads, tags, citations) and **curation** (many candidate versions, one wins per reader's lens). The case with *no clean story* is **convergent shared mutable state** (a Google-Docs-style doc five editors co-own, "everyone sees everyone's merged latest"). CRDTs were dismissed early and **that dismissal deserves a real adversarial re-test here specifically** — this is possibly the highest-value single question in the pass: *can CRDT-merge be a read-time lens-resolution variant (per-author signed op-TAGs, merge = a deterministic read-time fold over the op-set), or does it genuinely break "author = recovered signer"?* Note EFS is *already* a coarse CRDT (G-set revocation, LWW-register slots, confluent admission) — the lean is "keep the kernel a trivial CRDT, richer convergence is app-layer," untested against the wiki case. Prior art: **git** (what is the merge op, who signs the result?), **Datomic** (accretion-only as a collaboration substrate), **RDF named graphs** (the vocabulary for multi-speaker truth), **Automerge/Yjs**, and especially **dat/hypercore autobase** and **Perkeep** (the two closest, under-studied prior arts).

- **Versioning / history / snapshots / undo / time-travel.** History is *inherent* — nothing is ever destroyed. Per-slot version chains (`supersededBy`), undo = re-assert-the-prior-claim, checkpoints as a Datomic-style per-author as-of axis. Where it strains: **cross-object / directory-level snapshots** ("restore `/projects` to Tuesday") — a checkpoint covers one author's state, not a subtree across many authors under a lens; what does "restore the folder to Tuesday" even mean when the folder is a lens-reconciled union? No global time-travel (no portable global `t`; as-of is per-author or per-chain). Prior art: **Datomic** (`asOf/since/history` read vocabulary done right), **ZFS/btrfs COW** (the O(1) whole-subtree snapshot EFS structurally *can't* do), **git reflog**.

- **Move / rename at scale + symlinks / hardlinks / mount.** `movedTo` over immutable tagIds; because `tagId` derives from `parentTagId`, moving a 10k-child folder can't re-derive child ids — a "move" is a redirect at the folder node, children resolve under the old parent via the redirect. Walk it end-to-end: multi-hop `movedTo` composition (A→B→C), deep-link resolution through a moved parent, the `MAX_AUTO_FOLLOWS=8` budget on deep trees, path-compaction. Hardlinks are subsumed by "many PINs → one DATA" (better in one way — arbitrarily many first-class names; different in another — no refcount-GC, nothing is ever freed). Prior art: **Plan 9** (the deepest move/mount/namespace thinking ever done), **git** rename-inference-vs-explicit contrast, POSIX inode/dentry.

- **Trash / soft-delete / deletion — and the permanence tension.** Soft-delete is the native and *only* delete: "delete" = REVOKE the placement PIN → slot reads EMPTY, bytes persist, re-ASSERT to undelete (trash semantics + infinite undo, for free). **Hard delete does not exist and cannot** — this collides head-on with right-to-erasure / illegal content (deferred to a policy pass, but the FS pass keeps bumping into it) and needs to be *said* prominently as a known tradeoff. The only honest "truly gone" primitive on a permanent chain is **crypto-shredding** (encrypt, destroy the key) — a Pass-2 (privacy) dependency. "Delete for everyone" isn't expressible (it'd be a write-gate over others); community removal is **deny-shaped** (subtractive, per-reader), never destruction. The **WHITEOUT** reserved slot is the nearest cross-author tombstone — worth pinning what it can and can't promise.

- **Quotas / accounting / resource limits.** **Gas is the quota** — everyone pays their own writes, no shared pool, no quota daemon; a whole class of FS machinery simply doesn't exist because the economic layer meters it. `maxEntries` is a read-time filter, not a write-time cap (that would break replication-convergence). Whole-subtree accounting is a The-Graph/off-chain job (consistent with no query language). Separate, unsolved, and named out-of-scope: archive *sustainability* (who pays to keep 100-year bytes alive with no token) — a different question from write-metering.

- **Locking / concurrency / transactions / atomicity.** **Atomic multi-record writes are native and strong** (one signature over a Merkle root commits an arbitrarily large batch — stronger than POSIX). No write-write conflicts to lock against (separate author slots = optimistic-by-construction). Locks have no referent (reconcile at read time, not exclude at write time) — is there any app whose correctness *requires* write-time exclusion? Cross-author atomicity (Alice-and-Bob-both-sign-or-neither) is **not** expressible in one envelope — real FS need, or a chain-layer concern EFS correctly punts? Worth a plain consistency-model statement: EFS is per-venue-consistent, eventually-replicated, never globally-linearizable.

- **Multi-tag selection & the graph-query boundary.** Single-tag select is native (the per-tagId discovery index *is* a postings list). James wants **multi-tag AND** (tag A AND B AND C) *if cheap on-chain* — the scope doc's explicit ask is to **draw the line**: likely "bounded 2-3-tag AND over small containers is a view-contract convenience; anything unbounded, ranked, or NOT/OR is off-chain / The Graph." Pin *where* the boundary sits with a cost model — do **not** build a query engine. **EFS ships no query language** (export to a DB / The Graph / RPC); a real job here is *verifying* every FS operation (place/move/tag/revoke/supersede) stays cleanly subgraph-indexable via log-only-sync — more a verification task than a design one. Prior art: **The Graph** (the intended home — confirm the events map cleanly), postings-list intersection (roaring bitmaps / skip lists), Datalog/SPARQL as the explicit non-goal marker.

- **Naming / mounting / federation across containers.** Every author has a namespace root (their ADDRESS container); the container classifier resolves "what does this 64-hex word denote." **Cross-container mounting** — grafting author B's subtree into A's namespace — may be the most elegant unification available: Plan-9 union mounts (overlay several trees, first-hit wins) map *beautifully* onto first-attester-wins lenses. **Is "mount" just "lens," or a distinct spatial-grafting primitive?** Worth chasing. And the structural framing: **EFS has no single namespace; it has as many namespaces as there are lenses** — a federation feature, not a bug, that reframes "mount" and "discovery" entirely. Prior art: **Plan 9** (union mounts = lens resolution avant la lettre), **OverlayFS** whiteouts (the WHITEOUT slot is borrowed from here), ENS/IPNS (the Pass-3 human-name bridge).

- **Graph-DB-native features** — the half the "filesystem" framing under-serves; go wide here. Edges are first-class, typed, weighted, directional, author-attributed — a **property-graph substrate with signed, revocable, per-speaker edges**, and **natively reified / natively provenanced** (RDF's painful reification is EFS's default — every edge is already a signed claim with author, order, expiry, weight). Open white space: bounded **traversal** primitives (does the parent-walk generalize, or is all traversal an indexer job — same line as multi-tag?); **backlinks / inverse edges** ("cited by," "who mirrored this") — a small index-key-shape decision (does the discovery index key on edge *targets*?) with large app consequences; **reactive queries / watch / inotify** — EFS has *no push*, a pull/poll world; every real app needs "did anything under this path change," worth a blessed poll pattern even if the mechanism is off-chain; **schema / ontology** — almost none by design (string-only, permissionless edges), probably "schema is a read-side lens concern, never kernel-enforced," but say it because DB people look for `CREATE CONSTRAINT`. Prior art: **RDF/SHACL**, **Datomic** (the most convergent DB design — facts as 5-tuples with time), **TerminusDB/Dolt** (git-for-data), **Roam/Obsidian** (backlink UX).

- **The long tail** — xattrs (native, better: VAL reserved-key edges), content-addressed dedup (bytes dedup by CID; *identity* deliberately isn't content-derived — CAS people will be surprised), journaling/fsck (the chain is the journal; no corruption, but *availability* fsck — "are my mirrors alive?" — is a real different check), case/Unicode/reserved-names (NFC is enforced; confirm the full path-segment grammar, which is freeze-sensitive), timestamps (the `order`/`claimedAt`/`admittedAt` trio *is* the timestamp story — mtime untrusted, ctime trustworthy-per-chain, **atime doesn't exist** because reads leave no trace, a privacy feature). Most of these resolve to "native," "subsumed," or "explicitly out" — the value is in each having a *stated* answer so app developers hit an answer, not a silent gap.

## Cross-cutting notes to carry

- **The no-query-language / subgraph-indexability stance.** [settled non-goal] EFS does not and will not ship a query language. Devs export to a DB, The Graph, or RPC. The log-only-sync event set is designed to be subgraph-indexable; the pass's job at that boundary is to *confirm it holds* for every FS operation, and to draw the on-chain/off-chain line for multi-tag AND with a cost model — not to build query machinery.

- **The freeze-sensitivity rule.** [settled process] Deep design is staged (FS first), but **each pass must surface its freeze-sensitive reserved slots early** so the full "reserve before the ceremony" set from all three passes converges before the freeze. Staging the design does *not* stage the reservations. The most likely FS collision point is the frozen **record-kind table + reserved-key rows + derivation math**: any new FS edge type, sharing/permission record, versioning primitive, lock word, mount relation, or backlink index-key is either a reserved-key row (freeze-bound) or a user-key TAGDEF convention (free but un-vectored, per-client-dialect risk). Every row *not* minted should be an explicit "convention, not row" ruling, not silence. Several FS-shaped slots are *already* additively reserved and worth checking before inventing: **WHITEOUT** (≈ trash/cross-author tombstone), **salted/blinded TAGDEF** (≈ private folders, Pass-2-adjacent), **duplicate-member role**, **REF-to-property annotation role**, the **`successor`/`act`/delegation** slots (gate the delegated-authorship cluster). Read-side FS features (history views, restore UX, trash rendering, search over the index) are *Durable and cheap*; write-side features that need new record shapes or admission behavior are freeze-bound.

- **The OS handoff is input, not gospel.** [adjudicate, don't assume] The client-v2 round produced a pressure report (P1-P13) — explicitly material to *adjudicate*, not a spec, with only two red-team-confirmed findings. The FS-relevant asks: **P1** (store `admittedAt`, getProof-provable — the trustworthy-clock decision the timestamp and collaboration stories ride on), **P2** reserved-key candidates (`lang`/`dir`, handler-binding, freshness-beacon — each row-vs-convention-vs-reject), **P4** (no actor below the author → delegated write/revocation gap), **P7** (app-package-as-directory-tree, atomic resolve-at-pinned-root), **P9** (lens/view config must survive device loss — "a silent wipe of what a user *sees* is a truth bug"), **P10** (multi-device authorship, the 10 device bits, no allocation convention), **P13** (the timestamp-free-ID footguns and their blessed defenses — order by admission, cite the exact version, render edit history). Weigh these; don't inherit them.

## The sharpest open questions I deliberately did *not* answer

These are left open *for you* — not because they're unanswerable, but because answering them is the work, and a from-scratch look may dissolve or reframe them entirely. In rough order of where the good questions are densest, not of importance:

1. **Is the whole "access control" cluster mis-framed?** Should the pass retire "write permission" and replace it with "curated-view membership + delegated authorship + deny-subtraction," loudly?
2. **Can CRDT-merge be a read-time lens-resolution variant** (per-author signed op-TAGs, deterministic merge fold), dissolving the multi-writer strain — or does it genuinely break "author = recovered signer"? The early dismissal deserves a real adversarial re-test here. Probably the single highest-value question.
3. **Is "mount" just "lens"?** Union-directory federation and first-attester-wins are suspiciously isomorphic — unify them or cleanly separate them.
4. **What does "snapshot/restore the folder to Tuesday" mean** when the folder is a lens-reconciled union of many authors with per-author (not global) as-of?
5. **Where exactly is the multi-tag-AND (and bounded-traversal) on-chain/off-chain line**, with a cost model — and is it the *same* line for both?
6. **Does the discovery index key on edge *targets*, enabling native backlinks** ("cited by," "who mirrored this"), or is that off-chain? A small index-shape decision with large app consequences.
7. **Trace move-a-deep-subtree end-to-end** — does multi-hop `movedTo` composition, deep-link resolution through a moved parent, and the follow budget actually hold at 10k children?
8. **Is the time model right** — per-envelope vs per-record `order`; is `claimedAt` a blessed field or a convention; is the clean model bitemporal-plus-a-portability-patch; does `order` evaporate if transaction-time could travel? And should any of it still be called a "clock"?
9. **State the tradeoffs as first-class "what this gives up":** no hard delete (crypto-shred is a Pass-2 dependency), no global linearizability, mtime-is-untrusted. The P13 footguns in FS clothing.

And the meta-question over all of them: **which classic FS features are essential semantics to re-home, and which are artifacts of the one-mutable-cell architecture that should be declared gone rather than simulated?**
