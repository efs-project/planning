## What this rules

How a file **too large for one block** becomes **one signature**, with chunked/parallel/resumable submission behind that signature, permanent bytes on-chain, and forward-compatibility with future scaling. This is the "make the data bytes signatures too" idea taken to its full form: the user's single `eth_signTypedData_v4` transitively commits **every byte** of the file, and the chunks themselves carry **no signature** — each is admitted only if its bytes prove against the author-committed root.

The round designed three architectures and red-teamed each. **The three collapse to one genuine fork** (a fat/thin byte-layer dial): Architecture A and C are the same core differing only on Etched-sibling-contract (A) vs Durable-redeployable-store (C); B is A's core plus a blob transport rail plus a promotion lens (B ⊇ A). **No fatal flaw exists in the shared mechanism** — the envelope's already-fuzzed leaf verifier carries it.

## Recommendation: A-mechanism + B-lens + C-discipline

Ship **Architecture A** (native manifest + proof-streamed bytes, permanent tier-0 in state) with the mandatory red-team fixes, adopt **B's read/promotion lens**, and borrow **C's minimal-kernel discipline**.

### The mechanism

1. **One signed manifest.** The author signs the ordinary v2 envelope; one of its records is a **`chunks` reserved-key row on the file's DATA**, carrying a `FileManifest` body: `{chunksRoot, chunkCount, chunkSize, contentHash, size, encodingTag, preferredTier (advisory/non-binding), expiresAt}`. The kernel stores it **opaquely** (non-side-effecting — C's discipline; one row, no new entrypoint, no new kernel storage, `chunkSize` read at runtime, never a hard-coded 24576).
2. **`chunksRoot` commits every byte.** It is a keccak-Merkle root over the chunk hashes with the **chunk count bound at the apex** (Arweave `data_size` lesson) — only the true `n` reproduces the root, so a wrong count or a truncation/extension is a keccak second-preimage, not a trusted field. The commitment chain: `signature → recordsRoot → manifest leaf → chunksRoot → every chunk hash`. One signature, any file to ~100 TB (count is `uint32`), never forced to split.
3. **Bytes stream to a sibling Etched contract, no further signature.** `EFSBytes` (a separate ~200–350 LoC Etched contract — fills [[codex-kernel]]'s reserved second-artifact slot, keeps the kernel's EIP-170 budget untouched, and has zero coupling to kernel state) accepts `submitChunk(chunksRoot, tier, index, bytes, proof)` from **anyone** — admitted iff the bytes prove against the author-committed `chunksRoot`. `msg.sender` is ignored, exactly extending "anyone can relay a signed record" down to "anyone can relay a proven chunk." Storage key: `storeId = keccak(DOMAIN_CHUNKSTORE_V1, chunksRoot, tier)`.
4. **Two shipped tiers + off-chain mirrors.** **Tier 0 (SSTORE2, default):** bytes stored as contract code via `CREATE2(keccak(bytes))` — in the permanent state set, contract-readable via `extcodecopy` (paginated per EIP-7617), untouched by EIP-4444 history expiry, content-addressed so identical chunks dedup across files/chains. **Tier 2 (calldata):** cheapest; bytes ride in history (not state), archival ~1yr+, not contract-readable, commitment stays in state. Off-chain mirrors (`ipfs://`, `ar://`, `https://`) remain first-class. **Reserved, not shipped: tier 1 (inline SSTORE, storage-proof-native)** — its only distinguishing benefit, cross-chain storage proofs, overlaps EFS's own portability model (you *replicate* a file to the other chain and read it natively, rather than prove it across — the mechanism the substrate round chose over proofs), so it doesn't earn a distinct Etched mode in v2; **and tier 3 (blob)** — 18-day prune (see below). Tier numbering is preserved (1 and 3 reserved) so a future decision is additive, not a renumber.
5. **Resumable / parallel / idempotent — all for free under the one signature.** An on-chain **presence bitmap** (one word per 256 chunks) is the *stateless global upload session*: `missingChunks` returns the gaps, anyone resubmits only those (a *different* relayer can finish an abandoned upload), re-submit is a one-SLOAD no-op, chunks land out-of-order and in parallel (CREATE2 addresses are pre-computable → shard across M burners for M-fold throughput). Every gap is covered by the **original** signature — chunks never need one.
6. **Progressive reads, never broken.** An incomplete file reads as `BYTES-PARTIAL(k/n)` (computed, `n` is root-bound) — never a false "absent," never truncated-as-complete, never wrong-bytes-served (a lying store fails the proof → `CONTENT-MISMATCH`). Grades: `BYTES-COMPLETE@{STATE,EPHEMERAL,OFFCHAIN}`, `BYTES-PARTIAL(k,n)`, `BYTES-UNBOUND`, `CONTENT-MISMATCH`.

### What one signature honestly buys (the trilemma)

The red team confirmed the crypto core sound and the "one signature authorizes the whole upload" claim genuinely true **for authorization** — but sharpened what that means. One signature delivers **consent + content-identity + commitment-durability** (the `chunksRoot` in state is permanent and portable the instant the envelope lands). It does **not** by itself guarantee:
- **which permanence tier** the bytes land in — the *submitter* chooses the tier, and its incentive is the cheapest one. `{one signature, transparent no-resign promotion, author-bound permanence}` is a genuine trilemma — pick two. (Resolvable per-file with an *optional* signed `contractReadable` floor — ruled below.)
- **completion or funding** — the N chunk transactions still cost gas across N blocks; one signature says *who is allowed*, not *who pays*. An authorized-but-unfunded large upload's default outcome is an honest permanent `BYTES-PARTIAL`. Funding (relayer economics / bounty / escrow) is exogenous and unsolved — flagged by all three red teams.

**Base native-AA operational note (reviewed 2026-07-19):** EIP-8130's payer separation plus parallel/nonce-free transaction lanes may reduce coordination and sponsorship overhead for chunk workers on Base, but it changes none of the signed-manifest, proof, completion, or funding semantics above. Treat it as a versioned submission adapter and benchmark it on Vibenet; do not encode its nonces, payer, or transaction type into the manifest or `storeId`. See [[Reviews/2026-07-19-base-native-aa-impact]].

### Files, mirrors, and integrity — on-chain vs off-chain (ADR-0049 carried forward)

**A file's identity is its DATA record, not its bytes.** The v2 equivalent of v1's DATA attestation is a **DATA record** (`dataId = keccak(author, salt)` — owned, unforgeable, non-revocable, carrying no content): pure identity, a deterministic ID instead of an EAS UID. Everything else — retrieval locations (mirrors), content hash, size, the chunks manifest — hangs off it as edges. **On-chain and off-chain files are equally real files.** "On-chain" vs "IPFS" describes a file's *mirrors*, not the file; one DATA can carry several mirrors at once (a tier-0 store *and* an `ipfs://` *and* an `ar://`), same identity, multiple retrieval routes.

Integrity then splits by where the bytes live and how they are addressed — and **ADR-0049 is preserved throughout** (identity is never content-derived; off-chain hashes stay claims, never authenticated identity):

- **On-chain bytes (tiers 0/2):** *kernel-verified.* Each chunk is checked against the author-signed `chunksRoot` as it is submitted, so the stored bytes are provably the ones the author committed. Stronger than v1, where SSTORE2 bytes and the `contentHash` claim were never cross-checked.
- **Content-addressed off-chain mirrors (IPFS CID, Arweave data-root):** *transport-verified, for free.* The URI **is** a Merkle root of the content (IPFS UnixFS DAG / Arweave `data_root`), so a fetching client verifies the bytes against the CID itself — block by block, even for a 100 GB file. The chain never hashes it and does not need to; the author never re-hashes it (the CID they link **is** the hash). The residual is *"is this the CID the author meant"* — a lens-scoped trust claim, **not an integrity gap.** This is why the v1 "can't hash a 100 GB IPFS file" objection largely dissolves for content-addressed transports: the hash is the address, and Merkle-DAG verification happens at the transport layer at read time — the mirror image of EFS's own chunk verification at write time.
- **Location-addressed mirrors (`https://`):** *no transport integrity.* A `contentHash` claim is the only signal, and — exactly as ADR-0049 ruled — it is an **unverified, lens-scoped attester claim**, checkable only by a reader who downloads and hashes. The chain never verifies it. Useful as a *cross-transport binding* ("these mirrors all hold content with keccak = X"), still a claim.

So the manifest's `chunksRoot`/`contentHash` is a **verified commitment only for on-chain-submitted bytes**; for off-chain mirrors it is a claim (and largely redundant with a CID, which self-verifies). What Merkle adds is **not** off-chain hash verification — it is efficient one-signature commitment plus incremental on-chain verification for bytes that actually pass through the chain. Availability is graded honestly and never conflated with identity: on-chain bytes last as long as the chain (or its replicas); IPFS/Arweave bytes last as long as someone pins them / the endowment holds; the read layer labels each (`@STATE` / `@OFFCHAIN` / `@EPHEMERAL`) and never lets one masquerade as another.

### Forward-compat (rides the scaling curve, zero Etched change)

The signature commits chunk **hashes**, not a venue. Because tier is *not* committed and `storeId` includes it, bytes may live in several tiers and be **promoted up the permanence ladder by anyone** — re-proving each chunk against the *same* signed `chunksRoot`, no re-signature, reader-transparent. Bigger blocks / cheaper calldata → larger `chunkSize` (runtime, never hard-coded). EIP-7907 (24→64 KB code) → bigger tier-0 chunks automatically. keccak-Merkle stays the commitment of record (portable, ~12× cheaper than KZG, no trusted setup in the permanent path). The write-authorization design never changes again.

## Mandatory fixes (applied — from the red teams)

1. **DROP `submitChunkRun`** (the contiguous-run batch-proof optimization): novel, un-upgradeable, consensus-critical Merkle-batch crypto (OZ-multiproof CVE class) for ~5–15% on a tier where bytes already dominate cost — and the *entire* novel-crypto blast radius of `EFSBytes`. `submitChunk` reuses the envelope's already-fuzzed `verifyLeaf` and delivers every functional property. Removing it leaves `EFSBytes` with **zero novel crypto**. Highest-value single change.
2. **Scope reconstruction + portability pledges to tiers 0/1**, anchored on `EFSBytes`'s ERC-7201-frozen storage layout — *not* on a re-derived CREATE2 store address (red-team-C's F2: store initcode must evolve for bigger chunks, so the same `chunksRoot` yields different addresses across versions; anchor on frozen layout instead). Tier-2/3 bytes are archival/ephemeral and graded as such; the state-reconstructible 100-year pledge binds to tiers 0/1.
3. **No content-erasure for permanent bytes — say so.** Revoking the manifest hides the *pointer*, not the *content* (the `~store:<chunksRoot>` path serves bytes directly and never consults the manifest slot). The two "revocation solves this" / "spam is just gas" claims in the base text are false and corrected. → this forces James decision #2.
4. **Honest headline.** "One signature" advertises consent + content-identity + commitment-durability — not author-bound permanence, guaranteed completion, or a funded bill.

## Adopted from B (additive now, transport-independent)

B's **lens** wins unconditionally: the `BYTES-*` read grades incl. `EPHEMERAL-BYTES`; **permissionless keccak-verified promotion** over *any* channel or durable mirror (a promoter cannot corrupt the file — a wrong byte fails the root check); Filecoin-PDP-style mirror-liveness audit in the Durable read layer.

B's **blob rail is reserved, not shipped**, and `attestBlobPublication` is **dropped**. The red team found the blob transport dominated on every axis: blob bytes prune in 18 days (so "bank now, promote to L1 in 2030" is impossible — you must promote within 18 days at *today's* prices), no 2026 stock wallet builds a type-3 blob tx (breaking the self-submit-floor mission end for that rail), and a promoter re-keccak-verifies every chunk regardless of channel (so blobs buy nothing over "any channel + keccak-verify"). Ship tier-3 only when blob bytes become **durable** (EthStorage-class proof-of-storage), at which point it is a mirror tier A already supports — so deferring costs no rework.

## Etched vs Durable

**Etched (new):** the `chunks` reserved-key row + `FileManifest` body layout (one row in [[codex-kinds]]); the `EFSBytes` sibling contract — `DOMAIN_CHUNKSTORE_V1`, `storeId`, `submitChunk`/`submitChunks`, the chunk-Merkle construction (count-at-apex; frozen with golden vectors), ERC-7201 storage, tier constants (tier-0 = runtime code limit, **never** hard-coded). **Durable:** the `BYTES-*` grades + `readFile`/`missingChunks` views + `~store:` serving + tier-priority reduction ([[read-lens-spec]]); the promotion protocol; the SDK uploader/reassembler/promoter. **Reserved:** tier-3 blob + EthStorage target.

## James rulings (2026-07-07)

1. **Signed `contractReadable` floor — ADOPTED (renamed from `minTier`, James 2026-07-07).** The author may commit an optional boolean `contractReadable` requiring the bytes to be stored **on-chain, contract-readable (state tier)**. This is a **capability floor, not a tier number** — the author expresses the guarantee they want ("a contract must be able to read this"), the system picks the tier — which is a cleaner abstraction than exposing a tier enum (`minTier`), and generalizes: any future capability (e.g. cross-chain-provable → tier 1) is another named flag, not a number. It's the technically right name because contract-readability is uniquely **non-substitutable** — you cannot get it from Arweave/IPFS/anywhere off-chain, whereas permanence you can approximate off-chain (an Arweave endowment) — so the flag names exactly the property that *requires* this path. Read-enforced: a file with the floor set does not read as `COMPLETE` until its bytes exist in a contract-readable tier (you cannot force a submitter to *pay*, but the file never silently reads complete in a weaker tier). Default unset. **One documented caveat:** setting it also implies state-permanence + higher cost (in the EVM these are coupled — persistent contract-readability requires being in state, which is what makes it permanent); a "permanent but don't-care-about-contracts" author who won't pay for state uses an off-chain mirror instead. *Adjacent idea James raised (properties as forced-on-chain DATA): resolved not to merge — v2 already deleted the PROPERTY kind (values are interned on-chain data), and the interned-value vs owned-DATA split is principled — **content-address the verifiable** (small on-chain values → dedup, O(1) read) vs **owner-address the unverifiable** (files, possibly off-chain → author+salt); the same reason files aren't content-addressed. See [[codex-kinds]].*
2. **Fully permissionless byte pool — RULED (no gating).** "EFS is permissionless; anyone can write anything — that's what cypherpunk means; nobody, not even James, controls or runs it" (cf. arbitrary data already on Bitcoin). No manifest-gating of permanent tiers. **Consequences accepted, on the record:** permanent bytes are unattributed and non-retractable at the protocol level; there is no protocol takedown. Neutrality is preserved at the *protocol* layer; filtering lives entirely at the *edges* — **readers/gateways use lenses** (viewer sovereignty: write freely, read selectively) and **operators** make their own jurisdiction calls (operator-liability is an operator concern, not a protocol one; the operator-liability note stands). This *simplifies* the design — the permissionless pool ships as-is; no gating machinery to build.
3. **Frozen `EFSBytes`, dev-upgradeable → mainnet-immutable — RULED (matches James's framing exactly).** "Eventually one immutable contract everyone points to forever; during dev, upgradeable like the rest." This is the existing burn-to-immutable discipline (ADR-0048): `EFSBytes` runs behind a proxy on devnet, burns to immutable at freeze. **EIP-170** = the 24 KB limit on a single contract's *code* (not the file data); it's why byte-handling is a separate sibling contract from the kernel. The skeleton-compile is not a design choice — it just confirms `EFSBytes` fits under 24 KB as one contract; if it doesn't, we split it (still frozen). No further decision needed.
4. **L2/L3-first, L1 for the exceptional — RULED.** On-chain file bytes target L2/L3 as the primary, most-supported case (affordable); L1 is reserved for "extremely important historical documents." Note the scope: this is only about *on-chain file bytes* — **records/identities/metadata are chain-free and replicate freely; only large stored bytes are chain-located.**

   **Correction (2026-07-07): "replication is reserved/unbuilt" was an overstatement — the mechanism falls out of the core.** Replication = copy-pasting data onto another chain, nothing special: re-submit the chain-free signed records to chain B's kernel (author recovered from the signature → same IDs, permissionless, no author involvement) + for on-chain bytes, re-run `submitChunk` on B against the *same* `chunksRoot` (off-chain mirrors travel for free — an `ipfs://` CID resolves identically on any chain). This is a **direct consequence** of author-from-signature + chain-free IDs + content-addressed chunks — the same upload flow pointed at another chain. What is genuinely deferred is narrower and is *not* the mechanism: (a) **automated replicator tooling + LOCKSS "lots-of-copies" durability guarantees** (a UX/policy layer), and (b) **cross-chain "currency"** — knowing whether a replicated copy is the *latest/revoked* — which is the honestly-unsolved limit ([[efs-substrate-decision]] "not sold"), a *staleness* question, not a *can-you-copy* one. Given ruling #4 makes L2 the primary case and thus makes copies the durability hedge, priority items are: **run the portability validation** (below) and design the replicator/durability *tooling* — not "build replication," which the core already provides.
5. **Reserve the blob tier — RULED (A).** Blobs stay reserved until durable-blob tech (EthStorage-class) exists, at which point they're just another mirror A already supports. Promotion-funding economics are a later, exogenous track.

## The de-risking experiment (a few days, one L2 testnet)

One end-to-end vertical slice, excluding everything the red teams said to drop/reserve: chunk a ~1 MB file → build `chunksRoot` (count-at-apex) → **one** `eth_signTypedData_v4` over `{DATA, chunks-manifest, placement}` → a **relayer** submits the envelope then streams the ~43 tier-0 chunks via `submitChunk` **only** (no `submitChunkRun`) into a skeleton `EFSBytes` → **kill the relayer at 60%** → a **different** account reads the on-chain bitmap and finishes → read the whole file back via `extcodecopy`, check `isComplete`, verify end-to-end against `contentHash`. This single slice settles all five load-bearing uncertainties at once: (1) EIP-170 skeleton compile → the fat/thin dial; (2) CI gas snapshot on the real economic play (L2); (3) the shared crypto core end-to-end; (4) the most-contested property — one-signature + relayer + **resume-from-a-different-account-from-on-chain-state-alone**; (5) R1 readback (A's differentiator over C). Excludes `submitChunkRun`, blobs, promotion, `attestBlobPublication`.
